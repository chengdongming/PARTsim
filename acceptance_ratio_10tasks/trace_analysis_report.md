# PARTSim 追踪文件深度分析报告

**分析日期**: 2026-03-12
**分析对象**: `acceptance_ratio_10tasks/traces/` 目录下的 900 个 JSON 追踪文件
**分析目的**: 排查 Acceptance Ratio 图表中的两个异常凸起现象

---

## 问题背景

在进行 Acceptance Ratio（接受率）测试时，发现宏观图表出现了两处严重违背"负载增加，成功率单调下降"物理规律的异常凸起：

1. **U=0.7 的"反物理"凸起**: `ST-Sync` 算法在 U=0.5 和 0.6 时接受率已经降为 0，但在 U=0.7 时突然反弹凸起到 40% 的成功率。
2. **U=0.2 的"早发"凸起**: 多个算法在系统极度空闲的 U=0.1 时成功率只有 60%，反而到了 U=0.2 时上升到了 70%。

---

## 核心发现：仿真边界效应（Boundary Effect）

### 统计总览

| 分类 | 数量 | 占比 |
|------|------|------|
| ✅ 真正成功 | 50 | 16.7% |
| ⚠️ 边界效应假失败 | 113 | **37.7%** |
| ❌ 真正失败 (dline_miss) | 137 | 45.7% |

**关键结论**: 37.7% 的"失败"实际上是**任务在仿真结束前几毫秒到达，被调度但来不及完成**，被误判为失败。

---

## 异常一：U=0.7 "反物理"凸起分析

### 现象描述

ST-Sync 算法的接受率曲线出现反常凸起：
- U=0.5: 接受率 0%
- U=0.6: 接受率 0%
- U=0.7: 接受率 **40%** ← 异常凸起

### 详细数据

```
U=0.5 ST-Sync:
  真正成功: 0
  边界效应失败: 9
  真正失败(dline_miss): 1
  当前统计方法下的接受率: 0/10 = 0%

U=0.6 ST-Sync:
  真正成功: 0
  边界效应失败: 10
  真正失败(dline_miss): 0
  当前统计方法下的接受率: 0/10 = 0%

U=0.7 ST-Sync:
  真正成功: 4
  边界效应失败: 5
  真正失败(dline_miss): 1
  当前统计方法下的接受率: 4/10 = 40%
```

### 根本原因

**边界效应的随机性**导致不同利用率点的表现差异：

| 利用率 | 真正成功 | 边界效应失败 | 真正失败 | 接受率 |
|--------|----------|--------------|----------|--------|
| 0.5 | 0 | 9 | 1 | 0% |
| 0.6 | 0 | 10 | 0 | 0% |
| 0.7 | 4 | 5 | 1 | **40%** |

U=0.7 的任务集恰好**避免了在仿真末期产生未完成任务**，因此表现出更高的接受率。

### 证据：未闭合任务详情

**U=0.5 [000] 样本的未闭合任务**:
```
任务: task_3 @ 29994ms (仿真结束前 6ms 到达)
  Arrival事件: time=29994
  Scheduled事件: 1次
  End_instance: 无
  Dline_miss: 无

任务: task_8 @ 29998ms (仿真结束前 2ms 到达)
  Arrival事件: time=29998
  Scheduled事件: 1次
  End_instance: 无
  Dline_miss: 无
```

**U=0.6 [000] 样本的未闭合任务**:
```
任务: task_0 @ 29996ms (仿真结束前 4ms 到达)
任务: task_4 @ 29998ms (仿真结束前 2ms 到达)
```

这些任务被**调度了但仿真先结束**（仿真时间 30000ms），导致没有 `end_instance` 事件，被当前判断逻辑误判为失败。

### 仿真末期任务统计

| 利用率 | 样本 | 最后500ms到达 | 最后100ms到达 | 未闭合任务数 |
|--------|------|---------------|---------------|--------------|
| 0.5 | [000] | 38 | 7 | 2 |
| 0.5 | [001] | 47 | 10 | 1 |
| 0.5 | [002] | 33 | 5 | 3 |
| 0.6 | [000] | 25 | 6 | 2 |
| 0.6 | [001] | 30 | 5 | 1 |
| 0.6 | [002] | 33 | 6 | 1 |
| 0.7 | [000] | 31 | 6 | 3 |
| 0.7 | [001] | 19 | 3 | **0** |
| 0.7 | [002] | 31 | 6 | **0** |

U=0.7 的 [001] 和 [002] 样本没有未闭合任务，因此被判定为成功。

## 异常二：U=0.1/0.2 "早发"凸起分析

### 现象描述

多个算法在低利用率时出现接受率波动：
- U=0.1: 接受率 60%
- U=0.2: 接受率 70% ← 比 U=0.1 更高

### 详细数据

```
U=0.1 gpfp_st_sync:
  真正成功: 6
  边界效应失败: 4
  真正失败(dline_miss): 0
  接受率: 6/10 = 60%

U=0.2 gpfp_st_sync:
  真正成功: 7
  边界效应失败: 3
  真正失败(dline_miss): 0
  接受率: 7/10 = 70%
```

### 根本原因

**关键发现**: U=0.1 和 U=0.2 区间**没有真正的 deadline miss**，100% 的失败都是边界效应导致的！

接受率的波动完全由边界效应的随机性决定：
- U=0.1 恰好有 4 个样本在仿真末期有未完成任务 → 60%
- U=0.2 恰好只有 3 个样本在仿真末期有未完成任务 → 70%

开局能量状态（所有样本）：
```
能量: 20000 mJ = 20 J（与配置一致）
初始能量充足，不存在"开局暴毙"问题
```

## 根本原因总结

### 问题定位

| 现象 | 根本原因 | 是否 C++ 引擎 Bug |
|------|----------|-------------------|
| U=0.7 反常凸起 | 边界效应随机性 | ❌ 不是 |
| U=0.1/0.2 早发凸起 | 边界效应随机性 | ❌ 不是 |
| 仿真提前停止 | 无此现象 | - |
| Slack Time 溢出 | 无此现象 | - |
| 幽灵任务 | 无此现象 | - |

### 问题的真正来源

**Python 统计脚本的判断逻辑过于严格**：

当前逻辑（`TraceParser.get_acceptance_ratio()`）:
```python
# 当前逻辑：任何未闭合的 job 都判定为失败
if open_jobs:
    return 0.0
```

这导致仿真结束前几毫秒到达的任务（已调度但未完成）被误判为失败。

---

## 建议修复方案

### 方案一：修改判断逻辑（推荐）

修改 `TraceParser.get_acceptance_ratio()` 方法，**忽略仿真末期的任务**：

```python
def get_acceptance_ratio(self):
    """改进的二元可调度性判断 - 考虑仿真边界效应"""
    if not self.events:
        return 0.0

    open_jobs = {}
    has_arrivals = False
    has_dline_miss = False
    last_time = max(float(e.get('time', 0)) for e in self.events)

    for event in self.events:
        event_type = event.get('event_type', '')
        task_name = event.get('task_name')
        arrival_time = event.get('arrival_time')
        t = float(event.get('time', 0))

        if event_type == 'arrival':
            has_arrivals = True
            job_key = (task_name, str(arrival_time if arrival_time is not None else t))
            # 只记录仿真结束前 100ms 之前到达的任务
            if t <= last_time - 100:
                open_jobs[job_key] = True

        elif event_type == 'dline_miss':
            has_dline_miss = True
            job_key = (task_name, str(arrival_time if arrival_time is not None else t))
            open_jobs.pop(job_key, None)

        elif event_type == 'end_instance':
            job_key = (task_name, str(arrival_time if arrival_time is not None else t))
            open_jobs.pop(job_key, None)

    if not has_arrivals:
        return 0.0

    # 有 deadline miss 直接失败
    if has_dline_miss:
        return 0.0

    return 0.0 if open_jobs else 1.0
```

### 方案二：增加仿真时间

将仿真时间从 30000ms 增加到 60000ms 或更长，减少边界效应的相对影响：

```python
DEFAULT_SIMULATION_TIME = 60000  # 60秒仿真
```

### 方案三：修改任务生成器

确保任务不会在仿真末期到达，或者过滤掉末期到达的任务。

## 结论

1. **C++ 仿真引擎没有静默 Bug**：所有异常现象都是由边界效应导致的统计问题。
2. **Slack Time 计算正常**：没有发现溢出或异常跳跃。
3. **Deadline 检查器正常**：高利用率时确实会触发 deadline miss。
4. **问题在于统计逻辑**：需要修改 Python 脚本以正确处理仿真边界效应。
