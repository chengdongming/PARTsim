# ALAP 扩展调度算法详细设计文档 (Tick-based)

**文档版本**: v1.0
**适用架构**: Tick-based (1ms 步长) 实时调度仿真器
**设计目标**: 在 TIE/TGF/BTIE 的能量分发逻辑基础上，整合 ALAP (As Late As Possible) 时序机制，实现三种全新的调度策略。

---

## 1. 核心系统模型

### 1.1 时间与调度架构
* **调度范式**: **Tick-based (离散时间步长)**。
* **时间粒度**: $\Delta t = 1ms$。
* **触发机制**: 系统仿真主循环每推进 1ms，触发一次调度决策。所有状态（能量、Slack、死线）均在 Tick 边界处更新。

### 1.2 能量模型 (持续线性模型)
能量的收集与消耗在每个 Tick 期间是并发进行的。

* **更新公式 (Tick 结束时结算)**:
    $$E_{t+1} = \min(E_{max}, \ E_t + P_{harvest} \times 1ms - P_{consumed} \times 1ms)$$
* **准入检查 (调度决策时)**:
    判断当前电量是否足以支持接下来的 1ms 运行。对于功率为 $P_{task}$ 的任务：
    $$E_t + P_{harvest} \ge P_{task} \times 1ms$$

### 1.3 动态松弛时间 (Dynamic Slack)
这是 ALAP 机制的核心。对于就绪队列中的任务 $\tau_i$：

$$Slack_i(t) = AbsoluteDeadline_i - RemainingExecutionTime_i - CurrentTickTime$$

* **物理含义**: 如果不考虑抢占，任务最晚必须在 $t + Slack$ 时刻开始执行，否则将无法在截止期前完成。
* **ALAP 原则**: 只要 $Slack > 0$，说明时间还充裕，系统应当**强制休眠**以收集更多能量。

## 2. 算法详细逻辑设计

所有三个新算法都遵循 **"先检查时序(ALAP)，再检查能量(Distribution)"** 的两阶段处理流程。

### 2.1 算法一: ALAP-Block (基于 TIE 的阻断式扩展)

该算法复用了 TIE (Tick-based Instant Energy-aware) 的“死保高优”思想。

**逻辑流程 (每 1ms 执行):**

1.  **环境更新**: 收集本 Tick 的太阳能 ($E \leftarrow E + P_{harvest}$)。检查并移除已超时的任务。
2.  **阶段一: ALAP 时序门控**
    * 计算就绪队列中所有任务的 Slack。
    * 找出全局最小值: $S_{min} = \min(Slack_{all})$。
    * **判定**:
        * 若 $S_{min} > 0$: **强制休眠 (Force IDLE)**。本 Tick 不调度任何任务，直接结束，进入下一 Tick 充电。
        * 若 $S_{min} \le 0$: **唤醒**。进入阶段二。
3.  **阶段二: 能量分发 (TIE 逻辑)**
    * 按优先级 (RM) 排序就绪队列。
    * 取出最高优先级任务 $\tau_{top}$。
    * **能量检查**:
        * 若 $E_{curr} + P_{harvest} \ge P(\tau_{top})$:
            * **调度执行**: 运行 $\tau_{top}$ 1ms。
        * 若 **能量不足**:
            * **执行 BLOCK**: 本 Tick **强制空闲**。
            * **关键点**: 不尝试调度次高优先级的任务。因为 $\tau_{top}$ 已经到了必须运行的时刻 ($Slack \le 0$)，此时系统必须全力积攒能量给它，而不是把仅存的能量分给低优先级任务。
            * **后果**: $\tau_{top}$ 保持在队列首部，直到积累足够能量或彻底超时被移除。

### 2.2 算法二: ALAP-NonBlock (基于 TGF 的非阻断/贪婪扩展)

该算法复用了 TGF (Tick-based Greedy First) 的“贪婪填补”思想。

**逻辑流程 (每 1ms 执行):**

1.  **环境更新**: 收集太阳能。检查并移除已超时的任务。
2.  **阶段一: ALAP 时序门控**
    * 计算全局最小 Slack ($S_{min}$).
    * **判定**:
        * 若 $S_{min} > 0$: **强制休眠**。
        * 若 $S_{min} \le 0$: **唤醒**。进入阶段二。
3.  **阶段二: 能量分发 (TGF 逻辑)**
    * 按优先级 (RM) 遍历就绪队列中的任务 $\tau_k$。
    * **循环检查**:
        * **能量检查**: 若 $E_{curr} + P_{harvest} \ge P(\tau_k)$?
            * **YES**: 调度 $\tau_k$ 运行 1ms。**立即结束本 Tick 的调度**。
            * **NO**:
                * 如果 $\tau_k$ 是导致唤醒的任务 (即 $Slack_k \le 0$):
                    * **丢弃**: 判定 $\tau_k$ 无法挽救，**立即**从队列中移除 (Drop)。
                    * **Continue**: **继续循环**，检查下一个优先级的任务 $\tau_{k+1}$。
                * 如果 $\tau_k$ 的 $Slack_k > 0$:
                    * **Skip**: 跳过它，继续检查下一个任务。
    * **最终效果**: 如果高优先级任务没电跑不了，系统会贪婪地搜索后续能耗较小的任务来填补这个时间片，最大化处理器利用率。

### 2.3 算法三: ALAP-Sync (基于 BTIE 的同步批量扩展)

该算法复用了 BTIE (Batch TIE) 的“全有或全无”思想。

**逻辑流程 (每 1ms 执行):**

1.  **环境更新**: 收集太阳能。检查并移除已超时的任务。
2.  **阶段一: ALAP 批量时序门控**
    * **定义批次 (Batch)**: 取就绪队列中优先级最高的 $K$ 个任务 ($K \le 核心数$)。
    * 计算 **Batch Slack**: $S_{batch} = \min_{\tau \in Batch}(Slack_\tau)$。
    * **判定**:
        * 若 $S_{batch} > 0$: **强制休眠**。即使单个任务的 Slack 很小，只要 Batch 整体还能等，就一起等。
        * 若 $S_{batch} \le 0$: **唤醒**。进入阶段二。
3.  **阶段二: 能量分发 (BTIE 逻辑)**
    * 计算批次总能耗需求 (1ms): $P_{batch} = \sum_{\tau \in Batch} P(\tau)$。
    * **同步能量检查**:
        * 若 $E_{curr} + P_{harvest} \ge P_{batch}$:
            * **全员发射**: 调度 Batch 中所有任务运行 1ms。
        * 若 **能量不足**:
            * **整批阻塞 (Batch Block)**: 本 Tick **强制空闲**。
            * **关键点**: 一个任务都不能跑。系统认为这批任务是一个整体，必须同步执行。
            * **后果**: Batch 整体等待充电。如果等待时间过长，导致 Slack=0 的任务超时，该任务会在下一 Tick 的环境更新阶段被移除，Batch 将重组。

## 3. 算法特性对比

| 特性 | ALAP-Block | ALAP-NonBlock | ALAP-Sync |
| :--- | :--- | :--- | :--- |
| **基础逻辑源** | TIE (阻塞) | TGF (贪婪跳过) | BTIE (批量同步) |
| **ALAP 唤醒条件** | 全局任意 Slack $\le 0$ | 全局任意 Slack $\le 0$ | Batch 内任意 Slack $\le 0$ |
| **能量不足时行为** | **死锁 (Block)**: 停止一切调度，等待充电。 | **跳过 (Skip)**: 丢弃当前任务，尝试运行低优先级任务。 | **整批等待 (Wait)**: 所有任务暂停，等待充电。 |
| **优先级保证** | **强**: 严格遵守 RM，不惜超时。 | **弱**: 允许低优先级任务抢占资源。 | **强 (批次内)**: 保证批次同步性。 |
| **适用场景** | 任务极其重要，不允许乱序执行的场景。 | 追求高吞吐量，允许丢弃个别高耗能任务的场景。 | 并行依赖强，任务需要协同工作的场景。 |

## 4. 关键实现细节备忘

1.  **Tick 边界处理**: 所有的判断 ($Slack \le 0$, $Energy \ge P$) 都是基于 **Tick 开始时刻** 的状态。
2.  **能量计算**: 必须使用 `E + P_harvest` 进行准入判断，这反映了任务在执行过程中同时也在充电的物理事实。
3.  **死线丢弃 (Drop)**:
    * **ALAP-Block / ALAP-Sync**: 任务通常是在 Step 1 (环境更新) 时因为 `CurrentTime > Deadline` 被被动丢弃的。
    * **ALAP-NonBlock**: 任务可以在 Step 3 (分发) 时，因为 `Slack \le 0` 且无电，被**主动立即丢弃**，以便腾出资源给后续任务。
