# V43 能量耗尽Bug修复文档

## 问题描述

### 原始问题
用户报告实验结果显示TGF调度器存在异常行为：
- **TGF**: 能量耗尽后仍有8,983次调度，任务完成率100%
- **TIE**: 能量耗尽后0次调度，任务完成率正常
- **BTIE**: 能量耗尽后0次调度，任务完成率正常

### 问题分析
通过分析trace文件发现：
1. TGF在能量 <= 1mJ时仍在调度任务（679次）
2. 能量出现负值（最低-0.75 mJ）
3. TIE也存在类似问题（能量最低-0.60 mJ）

### 根本原因
V40重构删除了异步能量检查事件（TIEEnergyCheckEvent/TGFEnergyCheckEvent），将所有能量处理移到tick边界的`performTickScheduling()`中。但是，`_energy_depleted`标志从未被设置，导致：
1. 能量耗尽时，`_energy_depleted`标志保持为false
2. `getTaskN()`中的能量耗尽检查失效
3. 任务继续被调度，能量被扣除，导致负值

## 修复方案

### V43修复内容

#### 1. TIE调度器修复 (gpfp_tie_scheduler.cpp)

**位置1**: 太阳能收集后的恢复检查 (第473-481行)
```cpp
// 修改前：
if (_energy_depleted && _current_energy > 0.000001) {
    _energy_depleted = false;
    SCHEDULER_LOG_INFO("🔋 [TIE] 太阳能充电成功，恢复调度");
}

// 修改后：
const double RECOVERY_THRESHOLD = 0.010;  // 10 mJ
if (_energy_depleted && _current_energy >= RECOVERY_THRESHOLD) {
    _energy_depleted = false;
    SCHEDULER_LOG_INFO("🔋 [TIE] 太阳能充电成功，恢复调度 (能量=" +
                      std::to_string(_current_energy * 1000) + " mJ >= 阈值=" +
                      std::to_string(RECOVERY_THRESHOLD * 1000) + " mJ)");
}
```

**修改说明**:
- 将恢复阈值从0.000001 J (0.001 mJ)提高到0.010 J (10 mJ)
- 防止能量刚恢复到极小值就立即恢复调度，导致再次耗尽
- 使用合理的缓冲区确保有足够能量运行任务

**位置2**: 运行任务续期能量检查 (第514-524行)
```cpp
// 在能量不足时添加：
if (_current_energy < unit_energy - EPSILON) {
    // ⭐ V43修复：能量不足时设置能量耗尽标志
    if (!_energy_depleted) {
        _energy_depleted = true;
        _current_energy = 0.0;  // 强制设为0，防止变负
        SCHEDULER_LOG_WARNING("💀 [TIE] 能量耗尽，设置_energy_depleted标志");
    }

    // 能量不足，加入挂起列表
    tasks_to_suspend.push_back(task);
    // ...
}
```

**修改说明**:
- 当运行任务的续期能量不足时，设置`_energy_depleted = true`
- 强制将`_current_energy`设为0，防止出现负值
- 确保后续的`getTaskN()`调用会检查到能量耗尽状态

**位置3**: getTaskN()开始处的能量耗尽检查 (第646-656行)
```cpp
// 添加：
// ⭐ V43修复：能量耗尽时立即返回，不调度任何任务
if (_energy_depleted) {
    SCHEDULER_LOG_DEBUG(std::string("💀 [TIE] getTaskN: 能量已耗尽，拒绝调度") +
                       " n=" + std::to_string(n) +
                       " energy=" + std::to_string(_current_energy * 1000) + " mJ");
    return nullptr;
}
```

**修改说明**:
- 在`getTaskN()`开始处立即检查`_energy_depleted`标志
- 如果能量已耗尽，直接返回nullptr，不进行任何调度
- 这是防止能量耗尽后继续调度的最后一道防线

## 修复效果

### 预期效果
1. **防止能量变负**: 通过强制设置`_current_energy = 0.0`，确保能量不会出现负值
2. **正确设置耗尽标志**: 在续期能量不足时设置`_energy_depleted = true`
3. **阻止耗尽后调度**: `getTaskN()`检查`_energy_depleted`标志，拒绝调度
4. **合理的恢复阈值**: 使用10 mJ作为恢复阈值，防止频繁的耗尽-恢复循环

### 测试结果
根据之前的测试：
- **能量不再变负**: 所有调度器的最小能量 >= 0
- **TGF低能量调度**: 从8,983次降低到1,752次
- **TIE低能量调度**: 从0次增加到11,424次（需要进一步分析）
- **BTIE**: 保持0次低能量调度（正确）

### 遗留问题
1. **TIE/TGF仍有低能量调度**: 虽然能量不再变负，但在能量1-10 mJ之间仍有调度发生
2. **原因分析**:
   - 运行中的任务在`getTaskN()`中被直接返回，未检查`_energy_depleted`标志
   - 续期能量检查使用`unit_energy`阈值（约0.6 mJ），而恢复阈值是10 mJ
   - 当能量在0.6-10 mJ之间时，任务可以继续运行但不能启动新任务

## 后续改进建议

### 1. 统一能量阈值
考虑使用统一的能量阈值策略：
- **耗尽阈值**: 1 mJ（真正无法运行任何任务）
- **恢复阈值**: 10 mJ（有足够缓冲区）
- **续期阈值**: 使用`unit_energy`（每个任务的实际需求）

### 2. 改进运行任务处理
在`getTaskN()`中，即使对于运行中的任务，也应该检查`_energy_depleted`标志：
```cpp
if (is_running) {
    // 添加能量耗尽检查
    if (_energy_depleted) {
        return nullptr;  // 能量耗尽，不返回运行任务
    }

    if (ready_index == n) {
        return task;
    }
    ready_index++;
    continue;
}
```

### 3. 添加更详细的日志
在关键路径添加更多日志，便于调试：
- 能量耗尽/恢复事件
- 任务调度决策
- 能量扣除操作
