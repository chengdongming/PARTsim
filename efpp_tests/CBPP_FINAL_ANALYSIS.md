## 核心问题与解决方案

### 问题1：T=0调度延迟

**现象：**
修复前，第一次调度延迟到T=400ms才发生，导致所有任务第一次实例deadline miss。

**根本原因：**
1. T=0时，当前能量=0J
2. `canScheduleBatchWithEnergy()`使用前瞻性预测：0J + 15.642J - 0.301J > 0 ✅
3. 但`consumeBatchEnergy()`只检查当前能量：0J >= 0.301J ❌
4. 预扣失败，设置能量恢复事件
5. T=400ms后能量恢复，才成功调度

**解决方案：**
在`consumeBatchEnergy()`中添加"前瞻性信用"机制：
```cpp
// 如果当前能量不足，但前瞻性判断通过了，
// 允许使用"信用"进行预扣（能量允许暂时为负）
if (_current_energy < total_energy) {
    _current_energy -= total_energy;  // 允许为负
    // 预测的能量将在任务执行期间通过太阳能收集补充
}
```

### 问题2：能量恢复阈值

**修复前：**
```cpp
double energy_deficit = total_energy - _current_energy;
```
**修复后：**
```cpp
// 使用批量总能耗作为恢复阈值，确保能量足够支持整个批量
double energy_deficit = total_energy - _current_energy;
// 同时在日志中记录恢复阈值
SCHEDULER_LOG_INFO(" 恢复阈值=" + total_energy + "J");
```

### 问题3：批量清空机制

**添加了自动清空逻辑：**
```cpp
// 当请求索引n >= batch_size时，批量已全部dispatch，清空批量信息
if (_current_batch.is_scheduled && n >= _current_batch.batch_tasks.size()) {
    clearBatchSchedule();
}
```

## 最终算法流程（符合用户描述）

### 1. 批量决策阶段
```
K = min(空闲核心数, 未dispatch任务数)
选择前K个最高优先级任务
计算批量总能耗 = sum(energy_i)

max_wcet = max(WCET_i)  // 并行执行时间
预测收集能量 = getSolarIrradiance() × PV面积 × PV效率 × max_wcet

能量判断：当前能量 + 预测收集 >= 总能耗？
  ✅ 继续 | ❌ 设置能量恢复事件（阈值=总能耗）
```

### 2. 批量执行阶段
```
预扣批量总能量（使用前瞻性信用）
标记 _current_batch.is_scheduled = true
并行调度K个任务到K个核心
```

### 3. 批量完成阶段
```
当 n >= K 时，清空 _current_batch
触发下一次批量决策
```

### 4. 批量抢占阶段（已实现，未测试）
```
新任务到达时检查优先级
如果 新优先级 > 批量中某任务优先级：
  计算能量回退
  判断：当前能量 + 回退能量 >= 新批量能量？
  ✅ 执行批量重组 | ❌ 保持当前批量
```
