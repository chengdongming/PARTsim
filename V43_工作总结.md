# V43能量耗尽Bug修复 - 工作总结

## 问题回顾

**原始问题**：
- 用户报告TGF在能量耗尽后仍有8,983次调度，任务完成率100%
- TIE/BTIE在能量耗尽后0次调度（正常）
- 分析发现TGF能量出现负值（-0.75 mJ），TIE也有类似问题（-0.60 mJ）

**根本原因**：
V40重构删除了异步能量检查事件（TIEEnergyCheckEvent/TGFEnergyCheckEvent），将所有能量处理移到tick边界的`performTickScheduling()`中。但是，`_energy_depleted`标志从未被设置，导致能量耗尽时仍继续调度任务。

## 修复方案

### 三个关键修改

**1. 提高恢复阈值**（从0.001 mJ → 10 mJ）
```cpp
const double RECOVERY_THRESHOLD = 0.010;  // 10 mJ
if (_energy_depleted && _current_energy >= RECOVERY_THRESHOLD) {
    _energy_depleted = false;
}
```

**2. 设置能量耗尽标志并强制能量为0**
```cpp
if (_current_energy < unit_energy - EPSILON) {
    if (!_energy_depleted) {
        _energy_depleted = true;
        _current_energy = 0.0;  // 防止变负
    }
    tasks_to_suspend.push_back(task);
}
```

**3. 在getTaskN()开始处检查能量耗尽标志**
```cpp
if (_energy_depleted) {
    return nullptr;  // 拒绝调度
}
```

### 修改的文件
- `librtsim/scheduler/gpfp_tie_scheduler.cpp`
- `librtsim/scheduler/gpfp_tgf_scheduler.cpp`

## 附加发现

### 1. Trace文件对比

**文件大小差异**：
- TGF: 89 MB, 596,232 行（最小）
- TIE: 147 MB, 1,259,519 行（中等）
- BTIE: 213 MB, 1,352,897 行（最大）

**关键发现**：
- 三种算法的trace文件完全不同
- 虽然trace不同，但最终结果相同（任务完成数、能量消耗）
- 这证明了三种算法在能量充足的情况下达到了相同的效果

### 2. TGF为什么小那么多

**根本原因：DEBUG日志数量差异**

| 指标 | TGF | TIE | 差异 |
|------|-----|-----|------|
| 总DEBUG日志 | 199,337 | 774,364 | 3.9倍 |
| getTaskN DEBUG | 17,455 | 188,587 | 10.8倍 |
| std::cout数量 | 1个 | 6个 | 6倍 |

**具体原因**：
- TIE的getTaskN()有6个DEBUG输出，包括循环输出每个任务的详细信息
- TGF的getTaskN()只有1个DEBUG输出
- TIE在循环中输出导致日志量成倍增加
- 这是调试日志详细程度的差异，不是算法性能差异

### 3. 三种算法的调度策略差异

虽然最终结果相同，但调度过程不同：

**TIE (严格优先级阻断)**：
- 遇到能量不足立即停止级联
- 详细的能量扣除日志
- 日志量：中等

**TGF (贪婪填充)**：
- 跳过能量不足的任务继续扫描
- 简洁的日志
- 日志量：最少

**BTIE (批量门槛控制)**：
- 批量评估，全有或全无
- 详细的批量决策日志
- 日志量：最多
