【问题现象】
- BTIE: 22ms 正确中断 ✅
- TIE/TGF: 23ms 才中断 ❌ (应该在22ms中断)

【能量时间线】
  初始能量: 15.00 mJ
  前3个任务消耗: 12.83 mJ
  20ms时剩余: 2.17 mJ

  20ms: task_1调度 (arrival_time=20)

  21ms: 第1ms执行
        能量检查: 2.17mJ >= 0.6mJ ✅
        扣除后剩余: 1.57mJ

  22ms: 第2ms执行
        能量检查: 1.57mJ >= 0.6mJ ✅ (继续执行)
        扣除后剩余: 0.97mJ

        ⚠️ 问题：此时剩余0.97mJ，不足以支撑第3ms！
        应该在22ms检查时就中断，但实际继续了...

  23ms: 第3ms开始前
        能量检查: 0.97mJ >= 0.6mJ ✅ (仍然通过！)
        扣除后剩余: 0.37mJ
        任务实际执行0.x ms后能量耗尽中断

【根本原因】

TIE能量检查逻辑 (gpfp_tie_scheduler.cpp:154):

  if (current_energy <= unit_energy) {
      // 中断
  }

这个条件的问题是：
- 它检查的是"当前能量是否<= 1ms能耗"
- 但实际扣除是在检查**之后**进行的
- 如果扣除后剩余能量很少（如0.37mJ），任务会在下一个周期中异常中断

对比BTIE (gpfp_btie_scheduler.cpp:155):

  if (current_energy < unit_energy - EPSILON) {
      // 中断
  }

BTIE虽然也是类似的逻辑，但它使用了"预扣模式"，
在批量调度时已经扣除了能量，后续只检查不扣除。

【修复方案】

方案1: 提前检查（推荐）
  在检查时，判断扣除后是否有足够能量支撑下一次检查

  if (current_energy < unit_energy * 2 - EPSILON) {
      // 如果当前能量不足以支撑"当前1ms + 下一次检查的1ms"，提前中断
      // 这样可以避免任务在执行中途能量耗尽
      _scheduler->_kernel->suspend(_task);
      return;
  }

方案2: 改用严格小于
  if (current_energy < unit_energy - EPSILON) {
      // 使用严格小于，而不是<=
      // 这样当current_energy = unit_energy时也会中断
  }

方案3: 借鉴BTIE的预扣模式
  在调度时就预先扣除所有能量，而不是在运行时逐步扣除
