#### BTIE Bug详细分析

**现象**:
- BTIE在0能量时调度了3个任务
- task_1 (5ms WCET) - 完成✅
- task_2 (8ms WCET) - 完成✅
- task_3, task_4 也在运行

**日志证据**:
```
初始能量: 0.000000J
⚠️ notify: 能量不足 任务=task_1 需要=0.000558J 当前=0.000000J
⚠️ notify: 能量不足 任务=task_2 需要=0.000558J 当前=0.000000J
⚠️ notify: 能量不足 任务=task_3 需要=0.000558J 当前=0.000000J

但JSON追踪显示:
{ "time" : "5", "event_type" : "end_instance", "task_name" : "task_1"}
{ "time" : "8", "event_type" : "end_instance", "task_name" : "task_2"}
```

**根本原因**:
BTIE的`getTaskN()`函数**缺少能量检查**！

TIE的getTaskN()（正确）:
```cpp
double unit_energy = calculateUnitEnergyForTask(first_task);
if (_current_energy < unit_energy) {
    return nullptr;  // ✅ 能量不足，返回nullptr
}
return first_task;
```

BTIE的getTaskN()（错误）:
```cpp
// ❌ 没有能量检查！
if (ready_index == n) {
    return task;  // 直接返回任务，不管能量
}
```

### 3. 修复尝试 ⚠️

#### 尝试的修复
在BTIE的getTaskN()中增加了能量检查：
```cpp
double unit_energy = calculateUnitEnergyForTask(task);
if (_current_energy < unit_energy - EPSILON) {
    SCHEDULER_LOG_INFO("❌ 能量不足，无法调度");
    return nullptr;
}
```

#### 遇到的问题
修复后所有测试开始崩溃，可能原因：
1. 与performTickScheduling()的批量调度逻辑冲突
2. suspend()调用引发问题
3. 能量检查时机不当

#### 当前状态
已回滚修改，恢复到能量受限测试后的稳定版本
