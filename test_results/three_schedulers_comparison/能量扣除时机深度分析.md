# BTIE能量扣除时机深度分析

## 问题定义

### 当前BTIE实现

```cpp
void BTIEScheduler::performTickScheduling() {
    // 1. 收集所有任务
    std::vector<AbsRTTask *> all_tasks;

    // 添加运行中任务
    for (const auto& map_pair : running_tasks) {
        if (task) all_tasks.push_back(task);
    }

    // 添加新任务（K个）
    size_t K = std::min(free_cpus, _ready_queue.size());
    for (size_t j = 0; j < K; ++j) {
        all_tasks.push_back(sorted_ready[j]);
    }

    // 2. 计算总能耗（运行中 + 新任务）
    double total_energy = 0.0;
    for (auto* task : all_tasks) {
        total_energy += calculateUnitEnergyForTask(task);
    }

    // 3. 一次性扣除所有任务的能量
    if (_current_energy >= total_energy) {
        _current_energy -= total_energy;  // ← 预扣了新任务的能量！
        _batch_scheduled_this_tick = true;
        _current_batch_tasks = all_tasks;
    }
}
```

### TIE的实现

```cpp
void TIEScheduler::performTickScheduling() {
    // 1. 先扣除运行中任务的能量
    double total_energy_to_deduct = 0.0;
    for (const auto& map_pair : running_tasks) {
        AbsRTTask* task = map_pair.second;
        if (task) {
            total_energy_to_deduct += calculateUnitEnergyForTask(task);
        }
    }

    // 2. 扣除运行中任务的能量
    if (_current_energy >= total_energy_to_deduct) {
        _current_energy -= total_energy_to_deduct;  // ← 只扣除运行中任务
        _stats.total_energy_consumed += total_energy_to_deduct;
    }

    // 3. 进行新的调度决策（getTaskN会检查但不扣除新任务的能量）
}
```

## 关键问题：能量扣除的语义

### 什么是"扣除1ms能耗"？

**含义**：任务执行1ms所消耗的能量

**扣除时机有两种理解**：

#### 理解A：预扣（Budget-based）
- 在任务执行**前**扣除能量
- 语义：分配能量预算给任务
- 优点：能量不足时可以提前阻止任务执行
- 缺点：逻辑复杂，容易出现重复扣除

#### 理解B：后扣（Accounting-based）
- 在任务执行**后**扣除能量
- 语义：结算已执行任务的能耗
- 优点：逻辑清晰，符合实际
- 缺点：能量检查需要提前进行

### TIE采用哪种方式？

从代码来看，TIE采用**后扣方式**：

```cpp
// Tick 1ms开始时：
performTickScheduling() {
    // 扣除运行中任务的能量
    // 这些任务是上一tick执行到现在的，已经消耗了能量
    deduct(running_tasks);  // 结算上一ms的能耗

    // 然后进行新的调度
    getTaskN()会检查新任务的能量，但不扣除
}

// 新任务的能量何时扣除？
// 答案：在它们执行后的下一个tick扣除
```

**时间线**：
```
Tick 0: task_1, task_2开始执行
  └─ 不扣能量（刚开始）

Tick 1: task_1, task_2继续执行
  ├─ performTickScheduling()被调用
  ├─ 扣除task_1, task_2的能量（它们在Tick 0→1期间执行了1ms）← 后扣！
  ├─ getTaskN(0) → task_1（续期）
  └─ getTaskN(1) → task_2（续期）

Tick 2: task_1完成（执行了5ms），task_2继续
  ├─ performTickScheduling()被调用
  ├─ 扣除task_2的能量
  ├─ getTaskN(0) → task_2（续期）
  └─ getTaskN(1) → task_3（新调度，不扣能量）

Tick 6: task_2完成
  ├─ performTickScheduling()被调用
  ├─ 扣除task_3的能量（它在Tick 5→6期间执行了1ms）
  └─ ...
```

### BTIE当前采用哪种方式？

BTIE采用**预扣方式**：

```cpp
Tick 0: task_1, task_2到达
  ├─ performTickScheduling()被调用
  ├─ 收集all_tasks = [task_1, task_2]
  ├─ 计算total_energy = energy(task_1) + energy(task_2)
  ├─ 扣除total_energy ← 预扣了task_1, task_2的能量！
  └─ getTaskN()返回task_1, task_2

Tick 1: task_1, task_2继续执行
  ├─ performTickScheduling()被调用
  ├─ 收集all_tasks = [task_1, task_2]
  ├─ 计算total_energy = energy(task_1) + energy(task_2)
  ├─ 扣除total_energy ← 再次扣除！重复了！
  └─ ...
```

**问题**：
1. 预扣了新任务的能量
2. 每个tick都在扣除运行中任务的能量（重复扣除）

## 为什么测试结果一致？

在能量充足的场景下：
- TIE：每个tick扣除运行中任务能量
- BTIE：每个tick预扣所有任务能量
- **总量相同**（都完成了相同的任务），所以结果一致

但在**能量不足**的场景下：
- TIE：后扣，能量管理精确
- BTIE：预扣，可能导致过度保守或逻辑混乱

## 根本问题：设计理念冲突

### BTIE的设计意图

从注释看：
```cpp
// 2. 批量扣减能耗（一次性扣减k个任务的1ms能耗）
```

"一次性扣减k个任务的1ms能耗"这里的"k个任务"应该是什么？

**选项1**：k个新调度的任务
- 语义：预扣新任务的能量预算
- 问题：与TIE的后扣方式不一致

**选项2**：k个运行中的任务
- 语义：结算运行任务的能耗
- 合理：与TIE一致

**选项3**：k个所有要执行的任务（运行中+新）
- 语义：预扣所有任务
- 问题：预扣逻辑复杂，容易出错

### 当前BTIE的错误

当前实现是**选项3**，但这导致了：
1. 预扣新任务的能量
2. 每个tick重复扣除运行中任务的能量
3. 与TIE的逻辑不一致

## 正确的理解

BTIE的"批量"应该是指：
1. **批量决策**：一次性决定调度哪些任务（不是逐个决策）
2. **批量检查**：一次性检查是否有足够能量运行新任务
3. **批量标记**：一次性标记要执行的任务

但能量扣除应该遵循：
- **后扣方式**：只扣除运行中任务的能量
- **新任务不预扣**：让它们执行后再扣除

这才是真正的"批量TIE"：
- 批量决策（一次性决定）
- 能量后扣（与TIE一致）
