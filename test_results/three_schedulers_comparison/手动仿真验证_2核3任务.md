## TIE调度算法 - 手动仿真

### 算法特点
- **级联调度**: 能量不足立即停止，不继续调度后续任务
- **Tick级抢占**: 每ms检查是否需要抢占
- **每ms扣减能量**: 运行中任务每1ms扣除对应能耗

### 0ms时刻调度决策

**就绪队列**: [task_1(优先级20), task_2(优先级30), task_3(优先级40)]
**CPU核心**: 2个 (CPU0, CPU1)

#### Tick 0ms调度:
1. **能量检查** (假设能量充足)
2. **getTaskN(0)**: 返回task_1 (最高优先级)
   - 分配到CPU0
3. **getTaskN(1)**: 返回task_2 (次高优先级)
   - 分配到CPU1
4. **getTaskN(2)**: 返回task_3 (但无空闲CPU，进入就绪队列等待)

**调度结果**:
- CPU0: task_1 (运行)
- CPU1: task_2 (运行)
- 就绪队列: [task_3]

### 时间轴: 0-20ms

| 时间 | CPU0 | CPU1 | 就绪队列 | 说明 |
|------|------|------|----------|------|
| 0ms | task_1开始 | task_2开始 | [task_3] | 2个核心同时运行 |
| 5ms | task_1完成 | task_2运行(5ms) | [task_3] | task_1执行完5ms WCET |
| 5ms | task_3开始 | task_2运行(5ms) | [] | CPU0调度task_3 |
| 10ms | task_3运行(5ms) | task_2完成 | [] | task_2执行完10ms WCET |
| 10ms | task_3运行(5ms) | IDLE | [] | CPU1空闲 |
| 15ms | task_3运行(10ms) | IDLE | [] | task_3继续 |
| 20ms | task_3完成 | IDLE | [] | task_3执行完15ms WCET |

**0-20ms能��消耗**:
- task_1: 5ms × 单位能耗
- task_2: 10ms × 单位能耗
- task_3: 15ms × 单位能耗
- 总执行时间: 30ms (在2个核心上)
- 实际wall-clock时间: 20ms

### 20ms时刻 - 新任务到达

**事件**: task_1第2个实例到达 (周期20ms)

**就绪队列**: [task_1]
**运行中**: 无 (20ms时所有任务都已完成)

#### Tick 20ms调度:
- CPU0: task_1开始
- CPU1: IDLE
- 就绪队列: []

### 30ms时刻 - 新任务到达

**事件**: task_2第2个实例到达 (周期30ms)

**就绪队列**: [task_2]
**运行中**: CPU0上有task_1 (剩余时间: 20-30=10ms已完成，还有?需要重新计算)

实际上：
- 20ms: task_1开始执行
- 25ms: task_1完成 (WCET=5ms)
- 30ms: task_2到达时，CPU0空闲，CPU1空闲

### 40ms时刻 - 多任务到达

**事件**:
- task_1第3个实例到达 (40ms)
- task_3第2个实例到达 (40ms)

**就绪队列**: [task_1, task_3] (按优先级排序)
**CPU状态**: 都空闲

#### Tick 40ms调度:
- CPU0: task_1开始 (优先级更高)
- CPU1: task_3开始
- 就绪队列: []

## TGF调度算法 - 手动仿真

### 算法特点
- **贪心策略**: 能量不足时跳过当前任务，继续查找后续任务
- **级联调度**: 但会跳过能量不足的任务，继续调度后续任务
- **Tick级抢占**: 支持高优先级任务抢占

### 与TIE的主要区别

在**能量受限**的情况下，TGF会跳过能量不足的任务，继续查找后续可以调度的任务。

**示例场景**:
假设当前能量只能支撑1个任务执行1ms:

**TIE行为**:
- getTaskN(0): 返回task_1，能量充足，调度
- getTaskN(1): 检查task_2，能量不足 → **返回nullptr，停止级联**
- 结果: 只有task_1运行

**TGF行为**:
- getTaskN(0): 返回task_1，能量充足，调度
- getTaskN(1): 检查task_2，能量不足 → **跳过task_2**
- getTaskN(1): 继续检查task_3，如果task_3能耗更低（虽然优先级低但可能能耗不同）
- 结果: 可能调度task_1和task_3（贪心选择）

**注意**: 在当前测试中，如果三个任务的单位能耗相同，TGF的贪心策略不会体现出来。

## BTIE调度算法 - 手动仿真

### 算法特点
- **批量调度**: 一次性计算所有任务的总能耗
- **全有或全无**: 能量不足则不调度任何任务
- **批量扣减**: 一次性扣除k个任务的能耗

### 0ms时刻批量调度决策

**所有待调度任务**: [task_1, task_2, task_3]
**可执行任务数**: min(核心数=2, 就绪任务数=3) = 2

**批量决策** (performTickScheduling):
1. 收集所有需要执行的任务:
   - 运行中任务: 无
   - 就绪队列任务: task_1, task_2, task_3
   - 总计: 3个任务

2. 计算总能耗:
   - E_total = E(task_1) + E(task_2) + E(task_3)

3. 能量判断:
   - 如果 E_current >= E_total: 批量调度成功
   - 如果 E_current < E_total: **不调度任何任务**

4. **关键问题**: BTIE是调度所有3个任务，还是只调度2个（CPU核心数）？

根据代码分析（第418-434行）:
```cpp
// 收集所有需要执行的任务
std::vector<AbsRTTask *> all_tasks;
// 1. 添加运行中的任务
// 2. 添加就绪队列中的任务
// -> all_tasks包含所有任务（运行中+就绪）
```

**疑问**: BTIE会尝试调度所有任务，但只有2个CPU核心，这意味着什么？

### BTIE与TIE/TGF的关键区别

**TIE/TGF**:
- getTaskN()被kernel调用多次，每次返回一个任务
- kernel分配任务到空闲CPU
- 一次tick可能调度多个任务，但受CPU核心数限制

**BTIE**:
- 在performTickScheduling()中**一次性**批量决策
- 检查**所有任务**的总能耗
- 能量不足时，设置`_batch_scheduled_this_tick = false`
- 在getTaskN()中，如果`!_batch_scheduled_this_tick`，返回nullptr

**问题**: BTIE的"批量调度"是如何与"2个CPU核心"协调的？

## 关键问题分析

### 问题1: BTIE的批量调度范围

从代码看（第418-447行）:
```cpp
// 收集所有需要执行的任务
std::vector<AbsRTTask *> all_tasks;
// 1. 添加运行中的任务
const auto& running_tasks = _kernel->getCurrentExecutingTasks();
for (const auto& map_pair : running_tasks) {
    all_tasks.push_back(task);
}
// 2. 添加就绪队列中的任务
for (auto* task : _ready_queue) {
    all_tasks.push_back(task);
}
```

这意味着BTIE计算能耗时包括：
- **运行中任务**（需要续期）
- **就绪队列任务**（等待首次调度）

**批量决策**: 如果所有任务的总能耗 <= 当前能量，则批量调度成功

### 问题2: 能量受限时的行为差异

假设能量只能支撑1.5个任务执行1ms:

**TIE**:
- getTaskN(0): task_1，能量充足 ✓
- getTaskN(1): task_2，能量不足 ✗ → 停止
- 结果: task_1运行

**TGF**:
- getTaskN(0): task_1，能量充足 ✓
- getTaskN(1): task_2，能量不足 → 跳过
- getTaskN(1): task_3，如果task_3能耗更低...
- 结果: 可能task_1和task_3（贪心）

**BTIE**:
- 计算总能耗: E_total = E1 + E2 + E3
- 如果 E_current < E_total → 批量调度失败
- _batch_scheduled_this_tick = false
- getTaskN(0): 返回nullptr
- getTaskN(1): 返回nullptr
- 结果: **没有任务运行**（包括运行中任务也不续期！）

**这可能是BTIE的一个严重问题！**
