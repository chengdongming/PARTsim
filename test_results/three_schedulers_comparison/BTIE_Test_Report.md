# BTIE批量调度算法验证报告

## 测试概述

本次测试旨在验证BTIE（Batch Tick-based Instant Energy-aware）调度算法是否正确实现了"批量调度"特性：**要么全部调度，要么全不调度**。

## 测试配置

```yaml
CPU核心数: 3
任务数: 3个周期性任务（周期分别为20ms、30ms、40ms）
初始能量: 50mJ
仿真时间: 0:00（午夜，无太阳能）
仿真时长: 200ms
```

## BTIE算法分析

### 核心逻辑

```cpp
// 批量大小计算
batch_size = min(CPU核心总数, 就绪队列任务数);

// 批量能量计算
for (int i = 0; i < batch_size; i++) {
    total_batch_energy += unit_energy(task[i]);
}

// 批量决策
if (current_energy >= total_batch_energy) {
    // 能量充足：批量调度所有k个任务
    executeBatchScheduling(batch_tasks, total_batch_energy);
} else {
    // 能量不足：不调度任何任务
    skipBatchScheduling();
}
```

### 关键发现

#### 1. 批量大小受限的原因

在测试场景中，批量大小k始终是0或1，原因是：

- **任务调度流程**：任务到达 → 加入就绪队列 → 立即调度到CPU → 从就绪队列移除
- **结果**：就绪队列中很少有多个任务同时等待调度
- **影响**：`batch_size = min(3, ready_queue_size)` ≈ 1

#### 2. BTIE vs TIE的本质区别

虽然BTIE在本次测试中批量大小主要为1，但与TIE仍有本质区别：

| 特性 | TIE | BTIE |
|------|-----|------|
| **调度方式** | 级联调度（逐个判断） | 批量调度（一次性判断） |
| **能量不足时** | 立即停止后续任务 | 不调度任何新任务 |
| **能耗扣减** | 每ms逐次扣减 | 批量一次性扣减 |
| **决策时机** | 调度每个任务前 | 调度批量任务前 |

#### 3. BTIE的"全有或全无"逻辑

**理论设计**：
```
如果有k个任务需要调度：
- 能量充足 → 调度全部k个任务
- 能量不足 → 不调度任何任务
```

**实际表现**：
```
由于就绪队列中通常只有1个任务：
- k=1时的批量调度 → 退化为TIE的行为
- 但仍保留了批量判断的逻辑框架
```

## 结论

### BTIE算法实现正确性

✅ **BTIE的批量调度逻辑实现是正确的**

1. **批量判断逻辑正确**：一次性判断是否可以调度k个任务
2. **能量扣减正确**：批量调度时一次性扣减k个任务的能耗
3. **"全有或全无"正确**：能量不足时跳过批量调度

### 测试场景局限性

⚠️ **当前测试场景无法充分验证BTIE的批量调度特性**

原因：
1. 任务到达时间分散（0ms、10ms、20ms）
2. 任务被调度后立即从就绪队列移除
3. 就绪队列中很少有多个任务同时等待

### 改进建议

要充分验证BTIE的批量调度特性，需要：

1. **同时到达的任务**：创建多个任务同时到达的场景
2. **短任务**：使用WCET=1ms的任务，快速释放CPU
3. **更多任务**：增加任务数量，使就绪队列中同时有多个任务等待
4. **批量能量验证**：验证k>1时的总能耗计算和扣减

## 三种算法适用场景

### TIE (Tick-based Instant Energy-aware)
- **适用场景**：任务到达时间分散，能量充足
- **优势**：灵活调度，充分利用能量
- **劣势**：可能导致部分任务调度后能量不足

### BTIE (Batch Tick-based Instant Energy-aware)
- **适用场景**：任务批量到达，需要严格能量控制
- **优势**：保证批量任务要么全部执行，要么全部等待
- **劣势**：在任务分散到达时退化为TIE

### TGF (Tick-based Greedy First)
- **适用场景**：需要最大化任务完成数
- **优势**：贪婪策略，跳过能量不足的任务继续调度
- **劣势**：可能导致低优先级任务优先执行
