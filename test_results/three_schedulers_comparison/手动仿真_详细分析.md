## TIE调度算法 - 逐ms手动仿真

### 算法关键逻辑回顾
```cpp
// performTickScheduling() - 每ms执行一次
1. 收集太阳能
2. 检查并中断能量不足的任务
3. 检查抢占
4. 触发dispatch()

// getTaskN(n) - 级联调度
for (size_t i = 0; i < _ready_queue.size(); ++i) {
    if (ready_index == n) {
        if (available_energy < unit_energy) {
            return nullptr;  // ⚠️ 立即停止级联
        }
        _dispatching_tasks_total_energy += unit_energy;
        return task;
    }
    ready_index++;
}
```

## TGF vs TIE 关键区别分析

在**能量充足**的情况下，TGF和TIE的调度行为应该**完全相同**。

只有在**能量受限**的情况下，两者的区别才会体现：

### 场景：能量只能支撑1个任务执行

**TIE行为** (级联停止):
```
getTaskN(0): task_1, 能量充足 ✓ → 调度task_1
getTaskN(1): task_2, 能量不足 ✗ → 返回nullptr，停止
getTaskN(2): 不会执行
结果: 只有task_1运行
```

**TGF行为** (贪心跳过):
```
getTaskN(0): task_1, 能量充足 ✓ → 调度task_1
getTaskN(1): task_2, 能量不足 → 跳过
getTaskN(1): 继续检查task_3
结果: 可能调度task_1和task_3 (如果task_3能耗更低)
```

**注意**: 在当前测试中，三个任务的单位能耗相同（都是1.2 mJ/ms），所以即使能量不足，TGF也无法找到更低能耗的任务来替代。

## BTIE vs TIE/TGF 关键区别分析

### BTIE的批量调度逻辑

```cpp
// performTickScheduling() - 批量决策
std::vector<AbsRTTask *> all_tasks;
// 收集所有需要执行的任务
// 1. 运行中任务
// 2. 就绪队列任务

// 计算总能耗
double total_energy = 0.0;
for (auto* task : all_tasks) {
    total_energy += calculateUnitEnergyForTask(task);
}

// 批量判断
if (_current_energy >= total_energy) {
    // 批量调度成功
    _batch_scheduled_this_tick = true;
    _current_energy -= total_energy;
} else {
    // 批量调度失败
    _batch_scheduled_this_tick = false;
}

// getTaskN()
if (!_batch_scheduled_this_tick) {
    return nullptr;  // 不调度任何任务
}
```

### BTIE的关键问题

**问题1**: BTIE计算的是**所有任务**的总能耗，还是**调度任务**的总能耗？

从代码分析（performTickScheduling第418-447行）：
- BTIE收集 `all_tasks` 包括：运行中任务 + 就绪队列任务
- 计算这些任务的**总能耗**
- 如果能量充足，设置 `_batch_scheduled_this_tick = true`
- 然后触发 `dispatch()`

**但是**，`getTaskN()` 仍然按照CPU核心数限制调度任务：
- getTaskN(0): 返回第1个任务 → CPU0
- getTaskN(1): 返回第2个任务 → CPU1
- getTaskN(2): 返回第3个任务 → 但没有空闲CPU了

**疑问**: BTIE的"批量调度"是如何与"2个CPU核心"协调的？

### 在能量充足时

BTIE的行为应该与TIE/TGF相同，因为：
- `_batch_scheduled_this_tick = true`
- `getTaskN()` 正常返回任务
- 调度逻辑相同

### 在能量不足时

BTIE的行为**完全不同**：

假设能量只能支撑1.5个任务，但当前有3个任务需要调度：

**TIE**:
- getTaskN(0): task_1 ✓
- getTaskN(1): task_2, 能量不足 → 停止
- 结果: task_1运行

**BTIE**:
- performTickScheduling: 计算total_energy = E1+E2+E3
- 能量不足: `_batch_scheduled_this_tick = false`
- getTaskN(0): 返回 nullptr
- getTaskN(1): 返回 nullptr
- **结果: 没有任何任务运行！**

**这是BTIE的一个严重问题！** 它的"全有或全无"策略可能导致CPU空闲，即使能量足以支撑部分任务执行。
