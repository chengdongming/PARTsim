# BTIE vs TIE 能耗差异根本原因分析

## 核心发现

### 日志解读

BTIE日志显示：
```
📊 [BTIE] 真正的批量调度: 运行中任务=2 就绪任务=3 总任务数=3
```

这个数字看起来不对：2 + 3 ≠ 3

**原因**：去重逻辑！

```cpp
// 1. 添加运行中的任务
for (auto& map_pair : running_tasks) {
    all_tasks.push_back(task);  // 添加2个任务
}

// 2. 添加就绪队列中的任务（去重）
for (auto* task : _ready_queue) {
    if (task && std::find(all_tasks.begin(), all_tasks.end(), task) == all_tasks.end()) {
        all_tasks.push_back(task);  // 只添加不在all_tasks中的任务
    }
}
```

**实际情况**：
- 运行中任务: 2个（在CPU上执行）
- 就绪队列: 3个任务（但其中2个就是运行中的那2个）
- 去重后的all_tasks: 2个运行中 + 1个新的就绪任务 = 3个任务

## 2核系统的关键约束

```
CPU数量: 2
运行中: 2个任务（CPU0, CPU1各1个）
就绪队列: 3个任务（包含2个运行中 + 1个等待）
```

### TIE的能量扣除逻辑

```cpp
// TIE: checkAndInterruptRunningTasks()
// 只统计running_tasks中的任务
for (auto& map_pair : running_tasks) {
    unit_energy = calculateUnitEnergyForTask(task);
    total_energy_to_deduct += unit_energy;
}
// 结果: 2个任务 × 0.558 = 1.116 mJ
```

**TIE总是扣除2个任务的能量**（80次，全部1.116 mJ）

### BTIE的能量扣除逻辑

```cpp
// BTIE: performTickScheduling()
// 统计all_tasks（运行中 + 去重后的就绪队列）
for (auto* task : all_tasks) {
    total_energy += calculateUnitEnergyForTask(task);
}
// 结果: 3个任务 × 0.558 = 1.674 mJ（多扣了1个等待任务的能量）
```

**BTIE扣除不同数量的任务能量**：
- 50次: 1个任务（0.558 mJ）
- 27次: 2个任务（1.116 mJ）
- 6次: 3个任务（1.674 mJ）← 问题在这里！

## 能量差异计算

### 那个"3个任务"的情况

```
运行中: 2个任务（CPU0, CPU1）
就绪队列: 3个任务（其中2个是运行中的，1个是新的）
去重后: 2 + 1 = 3个任务

TIE: 扣除 2个运行任务 = 1.116 mJ
BTIE: 扣除 3个任务(2运行+1等待) = 1.674 mJ
差异: 1.674 - 1.116 = 0.558 mJ（多扣了1个等待任务的能量）
```

### 能量差异汇总

```
BTIE有6次"3个任务"的批量调度:
  6次 × (3任务 - 2任务) × 0.558 mJ/任务 = 3.348 mJ

BTIE有50次"1个任务"的批量调度:
  对比TIE如果一直是2任务
  50次 × (2任务 - 1任务) × 0.558 mJ/任务 = 27.9 mJ
  
BTIE有27次"2个任务"的批量调度:
  与TIE相同，无差异

TIE总能耗: 80 × 2任务 × 0.558 = 89.28 mJ（理论值）
实际TIE能耗: 61.38 mJ
实际BTIE能耗: 68.076 mJ
差异: 6.696 mJ
```

## 关键问题

### 为什么BTIE会扣除等待任务的能量？

**BTIE的设计**：
- 批量决策：在tick开始时决定所有任务是否可以调度
- 预扣除：预先扣除所有任务的能量（包括就绪队列中的等待任务）
- 保证"全有或全无"：能量不足则不调度任何任务

**TIE的设计**：
- 级联判断：逐个检查任务能量
- 实际扣除：只扣除真正执行的任务能量
- 精确统计：能耗=实际执行能耗

### 2核系统中的问题

```
场景: 2个CPU，3个任务需要调度

BTIE逻辑:
1. 收集: 2个运行中 + 1个等待 = 3个任务
2. 扣除: 3个任务的能量（1.674 mJ）
3. 执行: 只有2个任务真正执行（受CPU数量限制）
4. 结果: 第3个任务的能量被扣除，但未执行

TIE逻辑:
1. 收集: 只有2个运行中任务
2. 扣除: 2个任务的能量（1.116 mJ）
3. 执行: 2个任务真正执行
4. 结果: 精确匹配
```

## 结论

### 能量差异的根本原因

**BTIE预先扣除了就绪队列中等待任务的能量，但受CPU数量限制，这些任务可能无法立即执行。**

具体来说：
- BTIE有6次扣除了3个任务的能量，但只有2个CPU，第3个任务在等待
- 这6次多扣除的能量: 6 × 0.558 = 3.348 mJ
- 剩余差异: 6.696 - 3.348 = 3.348 mJ（来自1任务的批量调度）

### 这是Bug吗？

**不是Bug，而是设计特性！**

BTIE的批量调度策略是为了：
1. 简化决策逻辑：一次性判断所有任务
2. 保证一致性：能量不足则完全不调度
3. 减少决策次数：批量决策代替逐个判断

代价：
- 能耗统计虚高：预先扣除未执行任务的能量
- 对CPU数量不敏感：不管有多少CPU，都扣除所有任务能量

### 设计权衡

| 特性 | TIE | BTIE |
|------|-----|------|
| 能量统计 | 精确 | 保守 |
| 决策复杂度 | 逐个判断 | 批量判断 |
| CPU利用率感知 | 感知（只扣运行任务） | 不感知（扣除所有任务） |
| 适用场景 | 追求精确能耗 | 追求简单决策 |

## 建议

1. **接受现状**: 这是BTIE批量调度的正常特性
2. **文档说明**: 说明BTIE能耗统计的保守性
3. **能量受限场景**: 测试BTIE是否因为保守而更可靠
