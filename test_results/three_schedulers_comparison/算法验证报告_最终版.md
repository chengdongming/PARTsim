## 算法核心逻辑回顾

### TIE (Tick-based Instant Energy-aware)
```cpp
// 核心特点：级联调度，能量不足立即停止
getTaskN(unsigned int n) {
    for (size_t i = 0; i < _ready_queue.size(); ++i) {
        if (ready_index == n) {
            if (available_energy < unit_energy) {
                return nullptr;  // ⚠️ 立即停止级联
            }
            _dispatching_tasks_total_energy += unit_energy;
            return task;
        }
        ready_index++;
    }
}
```

### TGF (Tick-based Greedy First)
```cpp
// 核心特点：贪心跳过，能量不足跳过当前任务
getTaskN(unsigned int n) {
    if (available_energy < unit_energy) {
        // 跳过这个任务，继续查找后续任务
        for (size_t j = i + 1; j < _ready_queue.size(); ++j) {
            // 尝试找到能耗更低的后续任务
            if (next_available >= next_unit_energy) {
                return next_task;  // 贪心替换
            }
        }
        return nullptr;
    }
}
```

### BTIE (Batch Tick-based Instant Energy-aware)
```cpp
// 核心特点：批量调度，全有或全无
performTickScheduling() {
    // 收集所有需要执行的任务
    std::vector<AbsRTTask *> all_tasks;
    // 计算总能耗
    double total_energy = 0.0;
    for (auto* task : all_tasks) {
        total_energy += calculateUnitEnergyForTask(task);
    }

    // 批量判断
    if (_current_energy >= total_energy) {
        _batch_scheduled_this_tick = true;
        _current_energy -= total_energy;
    } else {
        _batch_scheduled_this_tick = false;  // ⚠️ 不调度任何任务
    }
}
```

### 关键发现

#### 1. 调度序列一致性 ✅

**验证结果**: 在能量充足的情况下，TIE、TGF、BTIE三种算法的**调度行为完全一致**。

**原因**:
- 能量充足时，TGF的贪心跳过逻辑不会触发（所有任务都能调度）
- BTIE的批量判断总是成功（总能满足总能耗）
- 因此三种算法退化为相同的RM优先级调度

#### 2. 能耗差异 ⚠️

**异常发现**: BTIE的总能耗比TIE/TGF高出约21.8%！

**分析**:
- BTIE Tick总次数: 112 (vs TIE/TGF: 101)
- 额外的11个Tick导致额外的能量消耗
- **可能原因**:
  1. BTIE在任务到达时触发了额外的批量调度（insert函数）
  2. BTIE的批量调度逻辑可能存在重复调度
  3. BTIE的能量扣除时机与TIE/TGF不同

**代码位置** (gpfp_btie_scheduler.cpp:836-842):
```cpp
void BTIEScheduler::insert(AbsRTTask *task) {
    // ...
    // ⭐ BTIE改进：任务到达时也触发批量调度
    if (_ready_queue.size() >= 1) {
        performTickScheduling();  // ⚠️ 额外触发tick调度
    }
}
```

这个设计导致BTIE在每次任务到达时都会触发一次额外的调度，增加了Tick次数和能耗。

#### 3. BTIE的批量调度行为

**观察日志**:
```
BTIE 批量调度: 运行中任务=2 就绪任务=1 总任务数=1
BTIE 批量调度: 运行中任务=2 就绪任务=2 总任务数=2
BTIE 批量调度: 运行中任务=2 就绪任务=3 总任务数=3
```

**问题**: BTIE计算能耗时，`all_tasks`包含了运行中任务+就绪任务，但这可能导致：
- 重复计算能耗（运行中任务已经扣除过能耗）
- 额外的Tick事件（每次任务到达都触发performTickScheduling）

## 算法正确性验证

### ✅ 验证通过项

1. **RM优先级调度**: 三种算法都正确地按周期优先级调度任务
2. **多核调度**: 正确地利用2个CPU核心进行并行调度
3. **任务切换**: 在任务完成时正确地切换到下一个就绪任务
4. **周期性任务**: 正确地处理周期性任务的多次到达
5. **能量计算**: 正确地计算每ms能耗（0.558 mJ/ms）

### ⚠️ 发现的问题

#### 问题1: BTIE的额外Tick事件

**现象**: BTIE的Tick次数比TIE/TGF多11次

**原因**: BTIE在任务到达时触发额外的performTickScheduling()

**影响**:
- 增加了不必要的能量消耗
- 增加了调度开销
- 可能导致不必要的能量检查

**建议修复**:
```cpp
void BTIEScheduler::insert(AbsRTTask *task) {
    // ...
    // 移除这个额外的调度触发，或者只在特定条件下触发
    // if (_ready_queue.size() >= 1) {
    //     performTickScheduling();
    // }
}
```

#### 问题2: BTIE的批量能耗计算逻辑

**疑问**: BTIE在performTickScheduling中计算所有任务的总能耗，但getTaskN()仍然按CPU核心数限制调度任务

**潜在问题**:
- 如果能量只能支撑1.5个任务，BTIE会完全不调度任何任务（包括运行中任务）
- 这可能导致CPU空闲，即使能量足以支撑部分任务执行

**测试建议**: 在能量受限的情况下（如initial_energy=3.6mJ），测试BTIE的行为

## 算法差异总结

### TIE vs TGF

**能量充足时**: 完全相同
**能量受限时**:
- TIE: 级联停止，遇到能量不足的任务后不再调度后续任务
- TGF: 贪心跳过，跳过能量不足的任务，继续查找后续可调度任务

**实际差异**: 在本测试中（所有任务单位能耗相同），TGF的贪心优势无法体现。

### TIE/TGF vs BTIE

**能量充足时**:
- 调度序列相同 ✅
- BTIE能耗更高（额外Tick事件） ⚠️

**能量受限时**（推测，需要进一步测试）:
- TIE/TGF: 部分任务运行（能量充足的前k个任务）
- BTIE: 可能完全不调度任务（如果总能耗超过初始能量）

## 测试结论

### 1. 算法正确性 ✅

在能量充足的情况下，三种调度算法的调度逻辑都是正确的，能够：
- 正确地按RM优先级调度任务
- 正确地利用多核并行执行
- 正确地处理周期性任务
- 正确地计算和扣除能量

### 2. 实现差异 ⚠️

BTIE的实现存在一些额外开销：
- 更多的Tick事件（112 vs 101）
- 更高的能耗（0.074772J vs 0.061380J）
- 额外的调度触发

### 3. 改进建议

1. **BTIE优化**: 移除或优化任务到达时的额外调度触发
2. **能量受限测试**: 需要进一步测试能量不足时的行为差异
3. **贪心策略测试**: 需要创建不同能耗的任务来测试TGF的贪心优势
