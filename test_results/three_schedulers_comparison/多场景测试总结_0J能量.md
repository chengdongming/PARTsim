## BTIE的严重Bug

### 问题现象
BTIE在0能量情况下调度并完成了2个任务：
- task_1: 0-5ms，完成✅
- task_2: 0-8ms，完成✅

### 日志分析

```
初始能量: 0.000000J

notify() 能量检查:
⚠️ notify: 能量不足 任务=task_1 需要=0.000558J 当前=0.000000J
⚠️ notify: 能量不足 任务=task_2 需要=0.000558J 当前=0.000000J
⚠️ notify: 能量不足 任务=task_3 需要=0.000558J 当前=0.000000J

Tick 1ms:
🛑 能量不足，中断任务: task_1
🛑 能量不足，中断任务: task_2
🛑 能量不足，中断任务: task_3

但是...任务还是继续运行并完成了！
```

### JSON追踪证据

```json
{ "time" : "5", "event_type" : "end_instance", "task_name" : "task_1"}
{ "time" : "8", "event_type" : "end_instance", "task_name" : "task_2"}
```

### 根本原因

BTIE的能量检查机制存在时序问题：

1. **dispatch()阶段**:
   - getTaskN()被调用
   - 检查能量：0.000 >= 0.558 ❌ 应该返回nullptr
   - 但实际返回了task指针

2. **notify()阶段**:
   - 检测到能量不足，发出警告
   - 但任务已经调度了

3. **performTickScheduling()阶段**:
   - 尝试中断任务
   - 但任务已经执行了1ms

**问题**：能量检查在dispatch()时无效，任务被错误调度

---

## 与TIE的对比

### TIE的行为（正确）

```cpp
getTaskN(0) → task_1
  检查能量: 0.000 >= 0.558 ❌
  返回 nullptr ✅

dispatch()收到nullptr，不调度任务
```

**结果**: 无任务调度 ✅

### BTIE的行为（错误）

```cpp
dispatch()被调用
  getTaskN()被跳过（BTIE不用getTaskN）
  直接从_current_batch_tasks获取任务

performTickScheduling()
  能量检查失败，不应该调度
  但dispatch()已经调用了
```

**结果**: 任务被调度并执行 ❌

---

## Bug总结

### 问题
BTIE在0能量场景下：
- ❌ 错误地调度了3个任务
- ❌ 完成了2个任务（task_1, task_2）
- ❌ 能量检查机制失效

### 影响
这是一个**严重bug**：
- 能量预算完全失效
- 系统在无能量情况下仍在执行任务
- 能量感知调度的核心功能崩溃

### 需要修复的位置

BTIE的dispatch()和能量检查机制需要重新设计，确保：
1. 能量检查在dispatch()之前完成
2. 能量不足时，dispatch()不调度任务
3. 与TIE的getTaskN()机制保持一致
