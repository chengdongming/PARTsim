# TIE/TGF/BTIE三种调度算法能耗差异深度分析

## 问题现象

| 算法 | Tick次数 | 总能耗 | 差异 |
|------|----------|--------|------|
| TIE  | 101 | 0.061380J | 基准 |
| BTIE修复后 | 101 | 0.068076J | +10.9% |

## 核心差异分析

### 1. 能量扣除时机

#### TIE能量扣除逻辑

```cpp
// gpfp_tie_scheduler.cpp:860-882
void TIEScheduler::checkAndInterruptRunningTasks() {
    // 1. 遍历所有运行中的任务
    for (auto &map_pair : running_tasks) {
        // 2. 计算该任务执行1ms所需的能量
        double unit_energy = calculateUnitEnergyForTask(task);
        total_energy_to_deduct += unit_energy;
    }
    
    // 3. 扣除所有运行中任务上一ms的能量
    if (total_energy_to_deduct > 0 && _current_energy >= total_energy_to_deduct) {
        _current_energy -= total_energy_to_deduct;
        _stats.total_energy_consumed += total_energy_to_deduct;
    }
}
```

**关键点**:
- ✅ 只扣除**运行中任务**的能量（已经在CPU上执行的任务）
- ✅ 不扣除**就绪队列任务**的能量（还未开始执行的任务）
- ✅ 能量扣除时机：任务真正执行了1ms后才扣除

#### BTIE能量扣除逻辑

```cpp
// gpfp_btie_scheduler.cpp:420-467
void BTIEScheduler::performTickScheduling() {
    // 1. 收集所有需要执行的任务
    std::vector<AbsRTTask *> all_tasks;
    
    // 1.1 添加运行中的任务
    const auto& running_tasks = _kernel->getCurrentExecutingTasks();
    for (const auto& map_pair : running_tasks) {
        all_tasks.push_back(task);
    }
    
    // 1.2 ⚠️ 添加就绪队列中的任务
    for (auto* task : _ready_queue) {
        all_tasks.push_back(task);
    }
    
    // 2. 计算所有任务的总能耗
    double total_energy = 0.0;
    for (auto* task : all_tasks) {
        total_energy += calculateUnitEnergyForTask(task);
    }
    
    // 3. ⚠️ 批量扣除所有任务的能耗（包括就绪队列）
    if (_current_energy >= total_energy) {
        _current_energy -= total_energy;
        _stats.total_energy_consumed += total_energy;
        _batch_scheduled_this_tick = true;
    }
}
```

**关键点**:
- ⚠️ 扣除**运行中任务 + 就绪队列任务**的所有能量
- ⚠️ 就绪任务还未执行，但能量已预先扣除
- ⚠️ 能量扣除时机：批量决策时就扣除，不管任务是否真正执行

### 2. 能量扣除次数统计

| 算法 | 能量扣除次数 | 平均每次能耗 |
|------|-------------|-------------|
| TIE  | 80次 | 0.767 mJ/次 |
| BTIE | 83次 | 0.820 mJ/次 |

**差异**: BTIE多扣除了3次

### 3. 问题场景分析

#### 场景: 2个CPU，3个任务等待执行

**时刻0ms**:
```
CPU状态:
  CPU0: 运行中 task_1
  CPU1: 运行中 task_2

就绪队列:
  [task_3]
```

**TIE的能量扣除**:
- ✅ 只扣除运行中的task_1和task_2的能量: 2 × 0.558 = 1.116 mJ
- ✅ task_3在就绪队列，不扣除能量

**BTIE的能量扣除**:
- ⚠️ 扣除task_1 + task_2 + task_3的能量: 3 × 0.558 = 1.674 mJ
- ⚠️ task_3还未开始执行，但能量已扣除

**下一tick (1ms)**:
- task_1执行完毕，task_3开始执行
- TIE: 扣除task_2和task_3的能量: 2 × 0.558 = 1.116 mJ
- BTIE: task_3的能量在上一tick已扣除，本tick可能不扣除（如果在running_tasks中）

### 4. 能量差异来源

#### 差异1: 就绪任务的预扣除

```
BTIE有83次批量调度，其中部分包含了就绪队列任务
TIE有80次能量扣除，只包括运行中任务

差异: 3次 × 平均2.232 mJ/次 = 6.696 mJ
```

#### 差异2: 批量调度的特性

BTIE的批量调度策略：
- 在tick开始时，一次性决定所有任务是否可以调度
- 预先扣除所有任务的能量（包括就绪队列）
- 这是"批量决策"的一部分，确保能量充足才调度

TIE的级联调度策略：
- 逐个检查任务的能量是否充足
- 只扣除真正执行的任务能量
- 能量不足时立即停止

### 5. 算法特性对比

| 特性 | TIE | BTIE |
|------|-----|------|
| **能量扣除对象** | 运行中任务 | 运行中 + 就绪队列任务 |
| **能量扣除时机** | 任务执行1ms后 | 批量决策时立即扣除 |
| **调度策略** | 级联判断 | 批量判断 |
| **能量不足行为** | 立即停止，不调度后续任务 | 整个tick不调度任何任务 |
| **能耗统计** | 实际执行能耗 | 预扣能耗（可能虚高） |

## 结论

### 是否有Bug？

**不是Bug，而是算法特性！**

BTIE的能耗略高（+10.9%）是由于其批量调度策略导致的：

1. **预扣除就绪任务能量**: BTIE在批量决策时就扣除就绪队列任务的能量
2. **批量调度特性**: 这是为了确保"全有或全无"策略的一致性
3. **设计权衡**: 牺牲少量能耗精度，换取批量决策的简单性

### 能量差异的合理性

从能量管理的角度看：
- **TIE**: 精确统计，只扣除实际执行的能耗
- **BTIE**: 粗略统计，预先扣除计划执行的能耗

在能量充足的场景下，这个差异（+10.9%）可以接受。
在能量受限的场景下，BTIE可能更保守（提前扣除能量，避免过度承诺）。

### 建议

1. **保持现状**: BTIE的能耗略高是其批量调度策略的正常结果
2. **文档说明**: 在文档中说明BTIE的能耗统计特性
3. **能量受限测试**: 测试低能量场景，验证BTIE的保守性是否带来优势
