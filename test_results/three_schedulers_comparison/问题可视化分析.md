## 根本原因总结

### 直接原因

BTIE的`insert()`函数在每个任务到达时都调用`performTickScheduling()`：

```cpp
// librtsim/scheduler/gpfp_btie_scheduler.cpp:836-842
void BTIEScheduler::insert(AbsRTTask *task) {
    Scheduler::insert(task);
    addToReadyQueue(task);

    // ⚠️ 问题根源：每个任务到达都触发批量调度
    if (_ready_queue.size() >= 1) {
        performTickScheduling();
    }
}
```

### 设计意图

**目标**: "消除1ms延迟" - 任务到达时立即调度，而不是等待下一个tick

**问题**:
1. 对于同时到达的多个任务，会触发多次重复调度
2. 大多数额外调度没有实际意义（队列为空或已在调度）
3. 导致显著的额外开销

### 与TIE/TGF的对比

| 特性 | TIE/TGF | BTIE |
|------|---------|------|
| 0时刻第一个任务 | 触发调度 | 触发调度 |
| 后续任务到达 | 不触发调度 | 触发调度 ⚠️ |
| Tick边界调度 | 每tick一次 | 每tick一次 |
| 额外调度 | 1次 (0ms) | 10-11次 |

## 修复方案对比

### 方案1: 完全移除额外调度（推荐）

```cpp
void BTIEScheduler::insert(AbsRTTask *task) {
    Scheduler::insert(task);
    addToReadyQueue(task);

    // 只在0时刻第一个任务到达时触发
    if (_ready_queue.size() == 1 && SIMUL.getTime() == Tick(0)) {
        performTickScheduling();
    }
}
```

**预期效果**:
- Tick次数: 112 → 101 (-10.9%)
- 总能耗: 74.77 mJ → 61.38 mJ (-21.8%)
- 调度行为: 不变

### 方案2: 防抖机制

```cpp
void BTIEScheduler::insert(AbsRTTask *task) {
    Scheduler::insert(task);
    addToReadyQueue(task);

    // 1ms内只调度一次
    if (!_batch_dispatch_scheduled ||
        SIMUL.getTime() >= _last_batch_dispatch + Tick(1)) {
        performTickScheduling();
        _batch_dispatch_scheduled = true;
        _last_batch_dispatch = SIMUL.getTime();
    }
}
```

**预期效果**:
- 减少部分额外调度
- 保留"快速响应"特性

### 方案3: 批量收集延迟调度

```cpp
void BTIEScheduler::insert(AbsRTTask *task) {
    Scheduler::insert(task);
    addToReadyQueue(task);
    _batch_dispatch_pending = true;
}

void BTIEScheduler::performTickScheduling() {
    // 正常tick调度...

    // 处理批量调度
    if (_batch_dispatch_pending) {
        // 只调度一次
        doBatchDispatch();
        _batch_dispatch_pending = false;
    }
}
```

**预期效果**:
- 合并同时到达的任务
- 减少重复调度
