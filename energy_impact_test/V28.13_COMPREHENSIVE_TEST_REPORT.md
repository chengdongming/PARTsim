## V28.13关键修复

### 1. 能量预留机制（Multi-core Cascade Scheduling）

**核心问题**: V28.12版本在多核调度时，所有getTaskN()调用都看到相同的能量值，导致能量检查失效。

**V28.13修复方案**: 实现`_dispatch_reserved_energy`预留机制
- 在`getTaskN(0)`时清零预留能量（新的dispatch轮次）
- 计算可用能量 = 当前能量 - 已预留能量
- 能量不足时立即返回nullptr（ASAP核心算法：立即停止）
- 预留能量用于返回的任务

**代码实现**（[gpfp_asap_scheduler.cpp:2763-2787](librtsim/scheduler/gpfp_asap_scheduler.cpp#L2763-L2787)）:
```cpp
// ⭐ V28.13修复：ASAP算法核心逻辑 - 使用预留能量实现多核级联调度
if (n == 0) {
    _dispatch_reserved_energy = 0.0;  // 新dispatch轮次，清零预留
}

double available_energy = current_energy - _dispatch_reserved_energy;

if (available_energy < unit_energy - 1e-6) {
    SCHEDULER_LOG_WARNING("⚡ [V28.13] ASAP能量不足，停止调度");
    return nullptr;  // ⭐ 立即停止，后续低优先级任务不调度
}

_dispatch_reserved_energy += unit_energy;  // 预留能量
```

### 2. Kernel支持部分调度

**核心问题**: Kernel的dispatch()期望填满所有CPU，但能量约束可能无法满足。

**V28.13修复方案**: 修改dispatch循环，允许部分调度
- 当getTaskN()返回nullptr时，停止调度
- 设置num_newtasks=0，优雅退出
- 不抛出异常

**代码实现**（[mrtkernel.cpp:310-332](librtsim/mrtkernel.cpp#L310-L332)）

**V28.12（调度8次）**:

**可能原因**:

- 需要确认这是否符合ASAP算法设计意图

**建议**: 需要进一步确认ASAP调度的预期行为：
