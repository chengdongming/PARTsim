# V28.8 并发竞态条件修复报告

## 📋 问题描述

### 原始问题
在 Test 1 测试中发现异常：初始能量 0.3J 理论上只能调度 3 次任务（3×0.1J），但实际结果显示 8 次调度。

### 根本原因：并发竞态条件 (Race Condition)

**问题代码** ([gpfp_cascade_scheduler.cpp:1771-1803](../librtsim/scheduler/gpfp_cascade_scheduler.cpp)):
```cpp
bool GPFPCASCADEScheduler::consumeEnergy(double energy_joules,
                                         const std::string &task_name) {
    // ❌ 无锁保护 - 存在竞态条件！
    double current_energy = getCurrentEnergy();  // 步骤1：读取能量
    if (current_energy < energy_joules) {        // 步骤2：检查能量
        return false;
    }
    if (_use_local_energy) {
        _local_energy -= energy_joules;          // 步骤3：扣除能量
        success = true;
    }
    return success;
}
```

**竞态条件场景**：
```
时刻T: 能量 = 0.3J

线程A (task_0):           线程B (task_1):
---------                 ---------
read() → 0.3J             read() → 0.3J  ⚠️ 都读到0.3J！
check(0.3 >= 0.1) ✓       check(0.3 >= 0.1) ✓
consume(0.3 - 0.1)        consume(0.3 - 0.1)
write() → 0.2J            write() → 0.2J  ❌ 最终0.2J，不是0.1J！

结果：两个任务都通过了检查，但能量只扣除了一次
```

### 证据

从日志可以看到竞态条件的明显证据：
```
50ms时刻，3个任务的能量检查都显示相同的值：
  task_0: 当前能量: 0.100050J
  task_1: 当前能量: 0.100050J  ⚠️ 相同！
  task_2: 当前能量: 0.100050J  ⚠️ 相同！

这证明它们是并发读取的，不是顺序执行的！
```

## 🔧 修复方案

### 方案选择：递归互斥锁 (std::recursive_mutex)

**为什么使用 recursive_mutex？**

在修复过程中发现存在**嵌套锁调用**：
```cpp
bool GPFPCASCADEScheduler::consumeEnergy(...) {
    std::lock_guard<std::mutex> lock(_energy_mutex);  // 第一次加锁

    double current_energy = getCurrentEnergy();        // ← 嵌套调用
    // getCurrentEnergy() 内部也尝试加锁：
    //     std::lock_guard<std::mutex> lock(_energy_mutex);  // 第二次加锁
    //     ❌ 死锁！同一个线程不能对同一个mutex加锁两次
}
```

**recursive_mutex 特性**：
- 允许同一个线程多次获取同一个锁
- 必须释放相同次数才能解锁
- 性能略低于普通 mutex，但在这里影响可忽略

## 📝 经验教训

### 1. 并发Bug的特征

**难以复现**：
- 只在特定时序下出现
- 多核系统更明显
- 日志输出可能改变时序（heisenbug）

**明显证据**：
- 多个线程读到相同的值
- 结果不符合逻辑（0.3J支持8次调度）
- 使用valgrind/helgrind可检测

### 2. 修复过程中的陷阱

**const成员函数的陷阱**：
```cpp
double getCurrentEnergy() const {
    std::lock_guard<std::mutex> lock(_energy_mutex);  // ❌ 编译错误
    // const成员函数不能修改mutable mutex
}
```

**死锁陷阱**：
```cpp
void consumeEnergy() {
    lock(_energy_mutex);
    getCurrentEnergy();  // 内部也尝试加锁
    // ❌ 死锁！
}
```

**解决方案**：
- 使用 `recursive_mutex` 允许重入
- 或移除const修饰符
- 或拆分函数（内部不加锁版本）

### 3. 测试并发代码的建议

1. **压力测试** - 多次运行，查看是否稳定
2. **工具检测** - 使用thread sanitizer、helgrind
3. **日志分析** - 查找异常的重复值
4. **单元测试** - 针对临界区编写测试
