**第一次dispatch()调用**:

**Step 1: getTaskN(0) - 调度task_0**


**Step 2: getTaskN(1) - 尝试调度task_1**
```
⚡ [V28.13] ASAP能量不足，停止调度: task_1 需要=0.100000J 可用=0.094821J (当前=0.194821J 已预留=0.100000J) (n=1)
⏰ [V28.14] 设置能量恢复定时器: 需要能量=0.005179J 收集率=1.008000 J/s 等待时间=5.137698ms 触发时间=7ms
```

**ASAP算法核心行为 - 立即停止原则**:
- 当前能量: 0.194821J
- 已预留能量: 0.100000J (task_0)
- 剩余可用: 0.194821J - 0.100000J = 0.094821J
- task_1需要: 0.100000J
- **能量不足 → 立即停止，不调度task_1, task_2, task_3**

**能量恢复定时器计算**:
```
需要的能量: 0.100000J - 0.094821J = 0.005179J
收集速率: 0.001008 J/ms (1.008 J/s)
等待时间: 0.005179J / 0.001008 J/ms = 5.137698ms
触发时间: 2ms + 5.137698ms ≈ 7ms
```

**Trace文件记录**:
```json
{"time": "2", "event_type": "scheduled", "task_name": "task_0"}
```

**结果**: t=2ms时刻，只有task_0被调度，task_1/2/3因为能量不足被跳过

**关键问题：为什么是2个任务而不是3个？**

从trace文件看，t=7ms时刻只调度了task_1和task_2，但没有调度task_3。这是因为：

1. **3个CPU限制**: 系统只有3个CPU
   - CPU0: 运行task_0（从t=2ms开始，50ms时间片）
   - CPU1: 运行task_1（t=7ms开始）
   - CPU2: 运行task_2（t=7ms开始）
   - task_3: 没有可用CPU，等待下一轮

2. **能量预留机制**: 每次getTaskN()调用都会预留能量
   ```
   task_0预留: 0.100000J
   task_1预留: 0.100000J
   task_2预留: 0.100000J
   总预留: 0.300000J
   剩余可用: 0.581874J - 0.300000J = 0.281874J
   ```

3. **多核级联调度**: ASAP算法在多核环境下，每个CPU调用一次getTaskN(n)
   - n=0 → task_0 (CPU0)
   - n=1 → task_1 (CPU1)
   - n=2 → task_2 (CPU2)
   - n=3 → 没有第4个CPU，不调用

## 为什么t=7ms调度多个任务？

### 根本原因

**t=7ms不是一个简单的时间点，而是新的调度轮次（dispatch round）**

### 详细解释

1. **t=2ms时刻**：
   - 能量: 0.194821J
   - 只够调度1个任务（0.1J）
   - ASAP原则：立即停止
   - 结果: 只调度task_0

2. **t=2→7ms期间**：
   - 持续收集太阳能
   - 5ms收集了0.487053J
   - t=7ms时能量达到0.581874J

3. **t=7ms时刻**：
   - 能量恢复事件触发
   - **启动新的dispatch轮次**（这是关键！）
   - 新的dispatch轮次中，能量足够调度3个任务
   - 但task_0已经在运行，所以只调度task_1和task_2

### 关键区别

| 项目 | t=2ms调度轮次 | t=7ms调度轮次 |
|------|-------------|-------------|
| 触发原因 | Kernel初始dispatch | 能量恢复事件触发 |
| 当前能量 | 0.195J | 0.582J |
| 可用CPU数 | 3个 | 2个（task_0占用1个） |
| 调度任务数 | 1个（能量不足） | 2个（受CPU限制） |
| ASAP行为 | 立即停止 | 继续调度直到填满CPU |

## 结论

### 为什么t=7ms调度多个任务？

**答案**: t=7ms不是普通的时间点，而是**能量恢复事件触发的新调度轮次**

**关键因素**:

1. **能量充足**: 0.581874J足够调度5个任务（0.1J each）
2. **CPU限制**: 只有3个CPU，task_0已占用1个，剩余2个CPU
3. **ASAP算法**: 在新的dispatch轮次中，按优先级调度task_1和task_2
4. **主动恢复**: V28.14的能量恢复机制自动触发新的调度轮次

### V28.14的核心改进

**从被动等待 → 主动触发**
- V28.13: 能量不足后，等待下一个事件触发调度
- V28.14: 能量不足后，计算等待时间，主动设置定时器触发新调度

这大大提高了能量利用率和调度效率！
