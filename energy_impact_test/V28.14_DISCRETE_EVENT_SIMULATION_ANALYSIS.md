# ASAP调度器V28.14 - 离散事件仿真机制详解

## 用户疑问分析

**问题**: 按照理论计算：
- t=3ms时，能量应该有 0.0974 × 3 ≈ 0.292J，足够调度task_1（需要0.1J）
- t=4ms时，能量应该有 0.0974 × 4 ≈ 0.390J，足够调度task_2（需要0.1J）
- 为什么实际都在 t=7ms 才调度？

**答案**: 这涉及到**离散事件仿真（Discrete Event Simulation）的时间推进机制**

---

## 核心问题：离散事件仿真的时间推进

### 什么是离散事件仿真？

MetaSim（我们使用的仿真引擎）是**离散事件仿真器**，不是连续时间仿真器。

**关键特性**:
1. **时间不是连续流动的**，而是跳跃的
2. **只在事件发生时推进时间**
3. **没有事件 = 时间停止**

### 仿真时间推进示例

```
t=0ms: 任务到达事件 → 处理 → 安排下一个事件
t=1ms: 等待队列检查事件 → 处理 → 安排下一个事件
t=2ms: 等待队列检查事件 → 收集能量 → 触发dispatch
t=7ms: 能量恢复定时器事件 → 触发新的dispatch
```

**关键**: t=2ms到t=7ms之间，**没有任何事件触发**，所以仿真时间直接从2ms跳到7ms！

## 实际时间线分析

### t=0ms: 任务到达

**���件**: 4个任务同时到达（arrival events）

**处理流程**:
```
1. task_0到达 → insert() → 能量检查 → 能量不足(0J) → 加入等待队列
2. task_1到达 → insert() → 能量检查 → 能量不足(0J) → 加入等待队列
3. task_2到达 → insert() → 能量检查 → 能量不足(0J) → 加入等待队列
4. task_3到达 → insert() → 能量检查 → 能量不足(0J) → 加入等待队列
```

**能量收集**:
```
第1次检查: 收集 0.097411J → 总能量 0.097411J
安排1ms后检查 (t=1ms)
```

### t=1ms: 第1次等待队列检查（定时事件）

**事件类型**: `ASAPWaitQueueCheckEvent`（定时器触发）

**处理流程**:
```
🔄 [V28.13] 检查等待队列（无新能量）: 当前能量=0.097411J
```

**关键点**:
- 这次检查**没有收集新能量**
- 只是检查等待队列
- 能量仍然不够（0.097J < 0.1J）
- 安排1ms后再次检查

**为什么没收集能量？**
这是V28.13的优化机制：避免频繁收集能量，只在特定时间点收集

### t=2ms: 第2次等待队列检查 + 第1次dispatch

**事件类型**: `ASAPWaitQueueCheckEvent` + `MRTKernel::dispatch()`

**处理流程**:
```
1. 检查等待队列:
   🔄 能量收集后检查等待队列: 收集=0.097411J 当前能量=0.194821J

2. 能量足够，触发kernel->dispatch():

   第1次dispatch轮次:
   - getTaskN(0): task_0 → 预留0.1J → 剩余0.0948J
   - getTaskN(1): task_1 → 需要0.1J > 可用0.0948J → ⚡ 能量不足，停止

3. 设置能量恢复定时器:
   ⏰ [V28.14] 设置能量恢复定时器:
     需要能量=0.005179J
     收集率=1.008000 J/s
     等待时间=5.137698ms
     触发时间=7ms
```

**关键点**:
- t=2ms收集了第2个0.097411J
- 总能量达到 0.194821J
- 调度了task_0，但能量不够调度task_1
- **设置了5.1ms后的定时器（t=7ms）**

### t=3ms, t=4ms, t=5ms, t=6ms: 没有任何事件！

**关键**: 这段时间**没有发生任何事件**！

**为什么？**
- t=2ms设置的能量恢复定时器触发时间是 t=7ms
- 在 t=7ms 之前，**没有其他事件被安排**
- 仿真引擎在 t=2ms 处理完所有事件后，发现下一个事件在 t=7ms
- **仿真时间直接从 2ms 跳到 7ms**

**能量收集的真实情况**:
```
虽然在物理世界中，t=3ms时能量应该是 0.0974 × 3 = 0.292J
但在仿真中，t=3ms这个时间点根本不存在！
仿真引擎直接从 t=2ms 跳到了 t=7ms
```

### t=7ms: 能量恢复事件 + 第2次dispatch ⭐

**事件类型**: `ASAPEnergyRecoveryEvent`（V28.14新增）

**处理流程**:
```
1. 能量恢复事件触发:
   🔋 [V28.14] 能量恢复事件触发 @ 7ms

2. 立即收集能量（一次性收集5ms的能量）:
   💰 [V28.14] 当前能量: 0.581874J
   (这是 t=2→7ms 共5ms收集的能量)

3. 触发新的dispatch轮次:
   第2次dispatch轮次:
   - getTaskN(0): task_0 → 预留0.1J → 剩余0.4819J
   - getTaskN(1): task_1 → 预留0.1J → 剩余0.3819J ✅
   - getTaskN(2): task_2 → 预留0.1J → 剩余0.2819J ✅
```

**关键点**:
- t=7ms是**第1个事件触发时刻**
- 从 t=2ms 到 t=7ms，仿真时间没有经过中间的任何时刻
- 能量是在 t=7ms 时刻**一次性收集**的，而不是持续收集

## 为什么不是 t=3ms, t=4ms 调度？

### 理论计算（连续时间）

如果时间是连续的：
```
t=2ms:  0.194821J (已收集)
t=3ms:  0.194821 + 0.097411 = 0.292232J ✅ 足够task_1
t=4ms:  0.292232 + 0.097411 = 0.389643J ✅ 足够task_2
```

### 实际仿真（离散事件）

```
t=2ms:  0.194821J (调度task_0，设置定时器)
t=3ms:  ⚠️ 这个时刻不存在！
t=4ms:  ⚠️ 这个时刻不存在！
t=5ms:  ⚠️ 这个时刻不存在！
t=6ms:  ⚠️ 这个时刻不存在！
t=7ms:  0.581874J (一次性收集5ms能量，调度task_1和task_2)
```

## 关键机制详解

### 1. 能量恢复定时器计算

**代码位置**: [gpfp_asap_scheduler.cpp:2840-2860](librtsim/scheduler/gpfp_asap_scheduler.cpp#L2840-L2860)

```cpp
// 计算需要的能量
double energy_needed = unit_energy - available_energy;  // 0.1 - 0.094821 = 0.005179J

// 获取收集率
double harvest_rate = EnergyBridge::getInstance().getHarvestingRate(adjusted_time);  // 0.001008 J/ms

// 计算等待时间
double wait_time_ms = energy_needed / harvest_rate;  // 0.005179 / 0.001008 = 5.14ms

// 设置定时器
MetaSim::Tick trigger_time = SIMUL.getTime() + wait_ticks;  // 2 + 5.14 = 7.14ms → 7ms
```

### 2. 离散事件仿真器的时间推进

**MetaSim的事件循环**:
```cpp
while (有未处理的事件) {
    Event* e = 获取最早的事件;
    SIMUL.setTime(e->getTime());        // ⭐ 直接跳到事件时间
    e->doit();                          // 处理事件
}
```

**示例**:
```
事件队列:
  - t=2ms:  WaitQueueCheckEvent
  - t=7ms:  EnergyRecoveryEvent
  - t=52ms: SlicingEvent (task_0时间片结束)

处理流程:
  t=2ms:  处理WaitQueueCheckEvent → 安排EnergyRecoveryEvent @ 7ms
  t=7ms:  处理EnergyRecoveryEvent  → 安排SlicingEvent @ 52ms
  t=52ms: 处理SlicingEvent
```

**中间时刻（t=3,4,5,6ms）不存在！**

## 能量收集的真实机制

### 连续收集 vs 离散收集

**物理世界（连续）**:
```
每1ms都在收集能量:
  t=0→1ms: 0.097J
  t=1→2ms: 0.097J
  t=2→3ms: 0.097J
  t=3→4ms: 0.097J
  ...
```

**仿真世界（离散）**:
```
只在事件触发时收集能量:
  t=0ms:   收集 0.097J (第1次检查)
  t=1ms:   不收集 (只是检查)
  t=2ms:   收集 0.097J (第2次检查 + dispatch)
  t=7ms:   收集 0.487J (5ms的能量一次性收集)
  t=52ms:  收集 4.383J (45ms的能量一次性收集)
```

### V28.14的能量恢复机制

**关键代码**: [gpfp_asap_scheduler.cpp:171-199](librtsim/scheduler/gpfp_asap_scheduler.cpp#L171-L199)

```cpp
void ASAPEnergyRecoveryEvent::doit() {
    // 1. 先收集能量（从上次事件到现在的所有能量）
    double harvested = _scheduler->updateEnergyContinuously(static_cast<TimeMs>(current_time));

    // 2. 获取当前能量
    double current_energy = _scheduler->getCurrentEnergy();

    // 3. 触发新的dispatch轮次
    kernel->dispatch();
}
```

**关键点**: `updateEnergyContinuously()` 会计算从上次收集时间到当前时间的**所有能量**，一次性累加。

## 为什么设置为5.14ms而不是3ms？

### 理论最优值

如果时间连续：
```
需要的能量: 0.005179J
收集速率:   0.001008 J/ms
理论等待:   0.005179 / 0.001008 = 5.14ms
```

但用户说：t=3ms时应该有 0.292J，足够调度task_1了

### 问题在哪里？

**t=2ms时刻的能量分配**:
```
总能量: 0.194821J
已预留: 0.100000J (task_0)
剩余可用: 0.094821J

task_1需要: 0.100000J
缺少: 0.100000 - 0.094821 = 0.005179J
```

**关键**: 系统不知道未来会收集多少能量，它只知道：
- 当前可用: 0.094821J
- 需要: 0.100000J
- 收集率: 0.001008 J/ms
- 计算: (0.1 - 0.094821) / 0.001008 = 5.14ms

**系统不会预测**"3ms后会有足够的能量"，它只是**按当前收集率计算需要等多久**。

## 总结

### 为什么不在t=3ms、t=4ms调度？

**答案**: **离散事件仿真的时间推进机制**

1. **t=2ms设置定时器**: 计算需要等待5.14ms
2. **t=3,4,5,6ms不存在**: 仿真时间直接从2ms跳到7ms
3. **t=7ms触发**: 这是定时器到期后的第1个事件时刻

### 能量收集的真实过程

```
t=2ms:  0.194821J (调度task_0，能量不足调度task_1)
t=2→7ms: 仿真时间跳跃（没有事件）
t=7ms:  0.581874J (一次性收集5ms能量，调度task_1和task_2)
```

### 关键机制

1. **离散事件仿真**: 时间只在事件发生时推进
2. **能量恢复定时器**: 按当前收集率计算等待时间
3. **一次性能量收集**: 在事件触发时刻收集从上次到现在的所有能量

---

## V28.14的设计权衡

### 优点

1. **简化实现**: 不需要每个ms检查能量
2. **减少事件**: 避免频繁的定时器事件
3. **准确预测**: 基于已知收集率计算等待时间

### 缺点

1. **不够精细**: 可能"等太久"（如t=3ms已有足够能量，但要等到t=7ms）
2. **预测误差**: 如果收集率变化，定时器可能不准确

### 改进方向

**动态调整策略**:
```cpp
// 不设置固定5.14ms定时器，而是每1ms检查一次
for (int i = 1; i <= 10; i++) {
    ASAPEnergyRecoveryEvent *e = new ASAPEnergyRecoveryEvent(this);
    e->post(SIMUL.getTime() + i);  // 每隔1ms检查一次
}
```

这样可以在t=3ms就触发调度，而不是等到t=7ms。
