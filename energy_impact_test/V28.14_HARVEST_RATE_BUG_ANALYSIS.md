# ASAP调度器V28.14 - 能量收集率异常分析

## 问题描述

在 t=2ms 时刻，能量恢复定时器计算等待时间时使用了错误的收集率。

### 观察到的异常

**日志显示**:
```
⏰ [V28.14] 设置能量恢复定时器: 需要能量=0.005179J 收集率=1.008000 J/s 等待时间=5.137698ms
```

**正确的理论值**:
```
8am辐照度: 541.2 W/m² (或接近值)
PV效率: 0.18
PV面积: 1.0 m²
收集功���: 541.2 × 0.18 × 1.0 = 97.416 W
收集率: 97.416 J/s = 0.097416 J/ms
```

**异常**: 日志显示 `1.008 J/s`，比正确值 **小了约97倍**！

### 验证实际的能量收集

从日志中的实际能量收集：
```
t=0→2ms: 收集 0.194821J
平均收集率: 0.194821J / 2ms = 0.097411 J/ms ≈ 97.411 J/s ✅
```

**结论**: 实际能量收集是正确的（~97.4 J/s），但 `getHarvestingRate()` 返回了错误的值（1.008 J/s）。

---

## 根本原因分析

### 可能的原因

1. **时间转换问题**: `getHarvestingRate()` 接收到的时间参数不正确
2. **缓存问题**: 返回了缓存的错误值
3. **单位转换错误**: 某处将 J/ms 错误地转换成了其他单位
4. **配置问题**: 仿真时间与太阳能数据时间不匹配

### 调试步骤

需要在代码中添加调试输出来确认：

## 对系统行为的影响

### 1. 等待时间计算错误

**错误的计算**:
```
需要能量: 0.005179J
错误收集率: 0.001008 J/ms (1.008 J/s)
计算等待时间: 0.005179 / 0.001008 = 5.14ms
```

**正确的计算**:
```
需要能量: 0.005179J
正确收集率: 0.097416 J/ms (97.416 J/s)
正确等待时间: 0.005179 / 0.097416 = 0.053ms ≈ 0.05ms
```

**影响**: 系统等待了 5.14ms，但实际上只需要等待 0.05ms！

### 2. 为什么仍然能正常工作？

尽管 `getHarvestingRate()` 返回了错误的值，系统仍然能正常工作，原因是：

1. **实际能量收集是正确的**:
   - `updateEnergyContinuously()` 使用的是正确的收集机制
   - t=2→7ms 实际收集了 0.487053J

2. **等待时间偏长，但不影响功能**:
   - 系统"等得太久"（5.14ms 而不是 0.05ms）
   - 但等了5ms后，能量肯定足够了
   - 只是调度延迟了约5ms

3. **离散事件仿真掩盖了问题**:
   - 仿真时间直接从 2ms 跳到 7ms
   - 中间的 t=3,4,5,6ms 不存在
   - 所以用户看不出系统"等太久"

## 为什么收集率是 1.008 J/s？

### 推测

1.008 J/s ≈ 0.001 J/ms，这个值接近配置文件中的 `base_harvest_rate_per_ms`。

让我检查配置：
```python
# energy_manager.py 中的默认值
base_harvest_rate_per_ms: float = 0.00002  # 0.02 J/s = 0.00002 J/ms
```

不对，0.00002 J/ms = 0.02 J/s，不是 1.008 J/s。

另一个可能是：
```
1.008 J/s = 0.001008 J/ms
```

这个值可能是某个计算错误的结果。让我检查是否有单位转换问题：

**可能性**: 某处将 W/m² 错误地当成了 J/ms:
```
错误计算: 541.2 W/m² / 537000 ≈ 1.008 J/s
```

但这个数字不对。

**更可能的原因**: 时间参数传递错误，导致 `getHarvestingRate()` 查询了错误的时间点（比如午夜或凌晨），返回了很小的收集率。
