## 🔍 CASCADE调度逻辑（正确理解）

### 核心算法

```
1. 任务按RM优先级排列: task_0 > task_1 > task_2 > task_3 > task_4

2. 从高到低检查每个任务:
   IF 任务在就绪队列 AND 能量 >= 单位能耗 THEN
       调度该任务1个时间片
       能量 -= 单位能耗
   ELSE
       跳过，检查下一个
   END IF

3. 如果任务WCET未完成，下一个时间片继续调度该任务
4. 如果所有任务都调度不了，等待能量恢复
```

### 关键点

1. **任务需要多个时间片完成WCET**
2. **能量充足时**: 高优先级任务优先调度，直到WCET完成
3. **能量不足时**: 跳过高能耗任务，调度低能耗任务
4. **WCET未完成**: 任务会持续占用CPU，直到完成或能量不足

## 🎯 关键发现

### 1. 为什么task_0,1,2在0ms和50ms都被调度？

**答案**: **它们的WCET未完成，需要多个时间片**

```
task_0: WCET=250ms，需要5个时间片
  0ms:   第1片
  50ms:  第2片 ← 继续调度
  100ms: 第3片 (能量不足，未执行)
  150ms: 第4片 (能量不足，未执行)
  200ms: 第5片 (能量不足，未执行)

结果: 只完成2个时间片，WCET未满足 ❌
```

### 2. 为什么100ms调度了task_3,4？

**答案**: **task_0,1,2能量不足无法继续，task_3,4在等待队列**

- 100ms时能量严重不足
- task_0,1,2 (bzip2, 0.1J) 能量不够
- task_3 (idle, 0.031J) 能量够，被调度
- task_4也可能通过能量回收被调度

### 3. 每次调度消耗多少能量？

**答案**: **取决于工作负载**

- bzip2: ~0.1J/时间片 (50ms)
- idle: ~0.031J/时间片 (50ms)
- 平均: ~0.04J/时间片

### 4. 0.3J为什么只能调度8次？

**能量消耗时间线**:
```
0ms:   调度task_0,1,2 (3×0.1J = 0.3J) → 剩余0J
50ms:  收集太阳能+能量回收，刚好够调度task_0,1,2
100ms: 能量很少，只能调度task_3(idle)
之后:  能量耗尽，系统停止
```
